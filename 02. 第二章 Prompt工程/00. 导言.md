# 第二章：Prompt 工程

![cover](image/prompt-engineering-cover.png)

**写 Prompt 不是填空，是一门手艺。**

很多人以为 Prompt 就是"给 AI 一句话让它做事"。真正用过的人知道：同样一件事，Prompt 写得好不好，输出质量可以差出一个量级。写得烂的 Prompt 得到的回答要么泛泛而谈，要么答非所问，要么干脆要你重新来过。

Prompt 工程解决的，就是**怎么把你的意图清晰、无歧义地传达给 AI**的问题。这听起来简单，但真正做起来你会发现：模型的"理解"和你的"表达"之间，总有一道鸿沟。Prompt 工程师的职责，就是架起这座桥。

本章从三个层次系统讲 Prompt 工程：

**第一层：基础概念**。理解 Prompt 的本质是什么，AI 处理 Prompt 的内部机制，以及为什么语言表达方式会显著影响输出质量。

**第二层：核心技法**。掌握主流的 Prompt 技术——从 Zero-shot 到 Few-shot，从 Chain-of-Thought 到 Tree-of-Thought，以及在 Claude 上的具体应用。

**第三层：实战设计**。学会针对不同任务类型（分析、代码、写作、推理）设计最优 Prompt，理解如何结构化组织 Prompt 内容，以及持续优化 Prompt 的方法论。

## 什么是"好的 Prompt"

一个好的 Prompt 具备三个特征：

**清晰（Clear）**：意思明确，没有歧义。你知道要什么，AI 也知道要什么。

**具体（Specific）**：边界清晰，范围明确。告诉 AI 做什么、不做什么、做到什么程度。

**结构化（Structured）**：信息组织有序，重点突出。AI 读起来不吃力，理解起来不出错。

反例："帮我看看这段代码有什么问题。"——代码在哪儿？什么问题？什么语言？什么场景？

正例："我有一段 Python 代码（见下方），在处理大文件时内存占用过高。请分析可能的原因，并给出优化建议，重点关注 IO 优化方向。"

区别在于：正例提供了具体的上下文、具体的约束条件、具体的评估标准。

## Claude 的 Prompt 特点

相比其他模型，Claude 在 Prompt 设计上有几个值得注意的特点：

**1. 更擅长遵循结构化指令**
Claude 对 XML 标签格式（如 `<input>` `</input>`）有专门的优化，在这种格式下的指令会比纯自然语言的指令更稳定地被遵循。

**2. 对"不要做"的指令比大多数模型更敏感**
明确说"不要做什么"比隐式假设更有效。Claude 会更认真地对待否定指令。

**3. 系统 Prompt 效果显著**
通过系统消息（System Prompt）设定角色和行为约束，比在每次对话中重复说明更高效。

**4. 长上下文理解能力强**
Claude 可以处理超长上下文，在设计 Prompt 时可以有更大的空间来提供示例和背景。

这些特点会影响具体怎么写 Prompt，在后面的文章中会逐一展开。
