# 第三节 Prompt 结构设计

## 信息怎么组织，AI 怎么理解

同样一段内容，用不同的顺序和结构组织，AI 的理解程度和输出质量可能完全不同。这一节我们深入讨论**如何结构化设计 Prompt**，让 AI 准确理解你的意图。

---

## Prompt 的通用结构

一个好的 Prompt，通常包含以下几个模块：

```
┌─────────────────────────────────────┐
│ 1. 角色设定（Role）                 │  ← 让 AI 进入专家模式
├─────────────────────────────────────┤
│ 2. 任务描述（Task）                 │  ← 清晰定义要做什么
├─────────────────────────────────────┤
│ 3. 上下文信息（Context）            │  ← 提供必要的背景
├─────────────────────────────────────┤
│ 4. 输出要求（Output Format）        │  ← 格式、长度、结构
├─────────────────────────────────────┤
│ 5. 约束条件（Constraints）          │  ← 边界条件和禁区
├─────────────────────────────────────┤
│ 6. 示例（Examples）                │  ← Few-shot 演示
└─────────────────────────────────────┘
```

### 1. 角色设定

给 AI 设定一个角色，可以显著改变它的输出风格和视角。

```python
# 没有角色
prompt = "写一段关于可持续发展的文字"

# 有角色
prompt = """
你是一位环保科学家，专注于可持续发展研究。
请写一段关于可持续发展的文字。
"""
```

有角色设定时，AI 会自动调用角色相关的知识框架、术语体系和表达风格。角色设定越具体，效果越好。

**常见有效角色**：
- 专业身份：资深软件工程师、市场分析师、律师
- 视角身份：一位40岁的企业管理者、一位初次接触这个领域的初学者
- 任务身份：担任会议记录员、担任代码审查员、担任面试官

### 2. 任务描述

任务描述必须清晰、具体、无歧义。

```python
# 模糊
prompt = "看看这段代码有什么问题"

# 清晰
prompt = """
请审查以下 Python 代码的性能问题：

[代码内容]

请关注：
1. 时间和空间复杂度
2. 可能的性能瓶颈
3. IO 优化方向

输出格式：
- 问题列表（按严重程度排序）
- 每个问题的原因分析
- 对应的优化建议
"""
```

清晰的任务描述 = **动作 + 对象 + 目标**。
- 动作：审查、分析、总结、翻译、生成
- 对象：Python 代码、这段文本、这份数据
- 目标：找出性能问题、提取关键信息、生成报告

### 3. 上下文信息

AI 没有你的背景知识，你需要提供必要的上下文。

```python
# 没有上下文
prompt = "这个方案有什么问题？"

# 有上下文
prompt = """
我正在为公司设计一个新的用户积分系统。
方案是：用户每消费1元获得1积分，100积分可兑换1元优惠券。

请分析这个方案可能存在的问题：
- 业务角度
- 技术角度
- 用户体验角度
"""
```

上下文信息不是越多越好，而是**与任务相关的信息**。无关信息会稀释 AI 的注意力，甚至引入干扰。

提供上下文的原则：
- **关联性**：只提供与任务直接相关的信息
- **简洁性**：去除冗余描述，保留核心信息
- **结构化**：用列表或段落组织，方便阅读

### 4. 输出要求

明确输出格式，让 AI 知道"长什么样"。

```python
prompt = """
请分析这份竞品报告，给出我们的竞争策略建议。

报告内容：[...]

输出要求：
## 竞品概况
（用表格呈现竞品对比：功能/价格/用户体验/市场占有率）

## 机会分析
（3-5 个我们可能的差异化方向）

## 策略建议
（分短中长期，各1-2条）

格式要求：
- 总字数控制在 1500 字以内
- 每个分析点都需要数据支撑
- 策略建议要可执行，避免空话
"""
```

输出要求包括：
- **结构**：用什么标题层级、是否用表格或列表
- **长度**：字数或段落数限制
- **风格**：正式/非正式、专业/通俗
- **内容要求**：必须包含什么、禁止包含什么

### 5. 约束条件

告诉 AI 什么不能做，往往比告诉它什么能做更有效。

```python
prompt = """
为我们的 AI 助手撰写新用户引导文案。

约束条件：
- 不要使用"革命性"、"突破性"、"首家"等夸张词汇
- 不要提及具体竞品名称
- 不要承诺做不到的功能
- 避免使用感叹号（！）超过 3 次
- 用户引导步骤不超过 5 步
"""
```

Claude 对否定指令比较敏感。明确说"不要"比隐式假设更有效。

### 6. 示例（Examples）

Few-shot 示例是控制输出质量最有效的方式之一。

```python
prompt = """
请按以下示例的风格和格式，回复客户的投诉邮件。

【示例 1】
客户投诉：订单延迟了一周还没到，货物也不全
回复：
尊敬的客户您好，
非常抱歉给您带来了不便。了解到您的订单延迟且货物不完整，我们高度重视。

经核实，您的订单因仓库调配原因延误，我们已于今日重新发货，并额外赠送您一张50元优惠券作为补偿。

如有任何问题，请随时联系我们。
此致

【示例 2】
客户投诉：收到的商品和描述不符，要求退货
回复：
尊敬的客户您好，
感谢您的反馈。对于商品与描述不符的情况，我们深表歉意。

我们可以为您提供以下方案：
1. 全额退款，商品无需退回
2. 免费换货，来回运费由我们承担

您希望选择哪种方案？
此致

【现在请回复】
客户投诉：[客户的投诉内容]
"""
```

示例的关键要素：
- **输入-输出对**：展示一个完整的情况和对应的正确回应
- **覆盖边界情况**：包括容易出错或混淆的例子
- **格式一致**：示例的格式就是期望的输出格式

## 信息顺序的艺术

### 位置效应

AI 对 Prompt 不同位置的内容，注意力权重不同。研究和实践表明：

- **Prompt 开头和结尾**的信息，AI 记忆最清晰
- **中间部分**容易被"稀释"
- **重要的约束条件**放开头或结尾效果更好

```python
# 效果可能不佳：约束在中间容易被忽略
prompt = """
请写一篇产品介绍文案。
产品是一款智能手表。
文案要专业且有吸引力。
不要使用夸张词汇（这条容易被忘记）
请突出健康监测功能。
这是给老年用户看的。
字数 500 字以内。
"""

# 效果更好：重要信息在开头和结尾
prompt = """
【关键约束】
- 用户是老年群体
- 重点突出健康监测功能
- 500 字以内
- 专业但通俗易懂
- 禁止使用夸张词汇

请写一篇产品介绍文案。

产品是一款智能手表，主要功能包括健康监测、紧急求助、简易操作。

请确保：
- 语言简洁，避免复杂术语
- 强调易用性和安全性
"""
```

### 结构化优于自然段落

当信息量大时，用**结构化格式**（列表、表格、XML标签）比自然段落更有效。

```python
# 自然段落（信息容易被忽略）
prompt = """
请为我们的产品写文案。产品是空气净化器，功能包括过滤PM2.5、静音运行、智能感应。目标用户是城市白领。用户痛点是城市空气质量差。品牌调性是简约现代。价格是2999元。字数300字。
"""

# 结构化（AI 更容易准确处理每个信息点）
prompt = """
## 产品信息
- 产品：智能空气净化器
- 价格：2999 元

## 核心功能
- 高效过滤 PM2.5
- 静音运行（<25dB）
- 智能感应自动调节

## 目标用户
- 城市白领
- 关注室内空气质量
- 有一定消费能力

## 品牌调性
- 简约现代
- 科技感
- 专业但不冰冷

## 文案要求
- 字数：300 字以内
- 突出：健康生活、品质感
- 避免：过度承诺、夸大效果
"""
```

结构化格式的优势：
- 每个信息点独立，不被其他内容稀释
- AI 可以按标签理解，不需要从自然语言中提取
- 避免信息遗漏或误解

### Claude 偏好的格式：XML 标签

Claude 对 XML 标签格式有专门的优化，在 `<input>` `</input>`、`<output>` `</output>`、`<context>` `</context>` 等标签下的内容，AI 会更准确地识别为指令或数据。

```python
prompt = """
<task>
请分析以下用户评论，找出产品的主要优点和缺点。
</task>

<data>
用户评论内容...
</data>

<output_format>
请用以下格式输出：
## 优点
-

## 缺点
-

## 改进建议
-
</output_format>
"""
```

在 Claude 的 API 中，系统提示（System Prompt）的内容和用户提示（User Prompt）的标签格式都遵循这个逻辑。

## 渐进式信息注入

对于非常复杂的任务，可以采用**渐进式**的信息提供方式——不要一次性把所有信息塞给 AI，而是根据 AI 的响应逐步提供。

```python
# 第一轮：确认理解框架
prompt1 = """
我需要你帮我做一个市场分析报告。
报告需要包含：
1. 市场规模和增长趋势
2. 主要竞争格局
3. 我们的机会和威胁
4. 策略建议

请先告诉我，你的分析框架是否完整？有没有需要补充的维度？
"""

# 第二轮：提供具体数据
prompt2 = """
基于你确认的分析框架，以下是我们掌握的数据：
[数据...]
请按照框架开始分析。
"""

# 第三轮：针对弱点补充
prompt3 = """
在上一轮分析中，机会部分的分析不够深入。
请重点补充：我们在技术优势方面有哪些具体机会？
"""
```

这种方式的优点：
- 确保 AI 理解正确再深入
- 允许中途调整方向
- 避免超长 Prompt 导致的注意力分散

## 本节小结

Prompt 结构设计的核心原则：

| 原则 | 说明 | 实践要点 |
|------|------|---------|
| 模块化 | 把 Prompt 拆成角色/任务/上下文/输出/约束/示例 | 每个模块独立，信息不混杂 |
| 结构化 | 用列表、表格、标签代替自然段落 | AI 对结构化信息处理更准确 |
| 位置意识 | 重要信息放开头或结尾 | 避免约束条件被稀释 |
| 渐进式 | 复杂任务分多轮 | 不要一次性给太多信息 |
| 格式标签 | 用 XML 标签区分指令和数据 | Claude 对标签格式有优化 |
