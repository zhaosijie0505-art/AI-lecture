# 第六节 实战案例

## 从需求到 Prompt 的完整过程

这一节我们通过两个完整的实战案例，展示如何从真实需求出发，设计、测试、优化 Prompt。

---

## 案例一：客服邮件回复生成器

### 需求背景

你的公司是一个 SaaS 产品，客户支持团队每天需要处理大量用户投诉和咨询邮件。当前的问题是：
- 不同客服人员写的回复质量参差不齐
- 新员工不知道如何写专业、友好的回复
- 回复时间过长，影响客户满意度

目标：做一个 Prompt，帮助生成标准化的专业回复。

### 第一步：分析需求

```
问题类型：写作任务（邮件回复）
核心挑战：
1. 需要理解客户的问题类型
2. 回复需要专业但友好
3. 不同类型问题需要不同的处理方式
4. 涉及公司政策的，需要遵循特定规则

约束条件：
- 回复必须包含特定元素（致歉/解释/解决方案）
- 不能做出未授权的承诺
- 需要识别何时需要升级
```

### 第二步：设计初版 Prompt

```python
prompt_v1 = """
请帮我写一封回复邮件。

客户问题：[问题内容]
"""
```

**评估**：这个 Prompt 太简单，输出一定不够用。

### 第三步：优化 Prompt

```python
prompt_v2 = """
你是公司客服，帮我写一封回复邮件。

客户问题：[问题内容]

要求：
- 专业、友好
- 解决问题
"""
```

**评估**：好一些，但仍然不够具体。没有格式要求，没有边界判断。

### 第四步：继续优化

```python
prompt_v3 = """
你是公司客服，需要回复客户的邮件。

【客户邮件】
[邮件内容]

【公司政策】
- 可以提供：退款、更换、优惠券、技术支持
- 不可以：承诺超出政策范围的补偿、承认法律责任
- 升级条件：涉及金额 > 1000元、或涉及法律问题、或客户表示将投诉监管部门

【回复要求】
1. 开头：称呼 + 感谢反馈 + 确认理解的问题
2. 中间：解释原因（如适用）+ 具体解决方案
3. 结尾：询问是否满意 + 提供联系方式

【语气】
专业但有温度，不冷漠不套话

输出格式：
---
主题：[邮件主题]

正文：
[邮件正文]
---
"""
```

**测试**：用这个 Prompt 测试了几封真实邮件，效果不错。

### 第五步：发现问题并修复

**问题**：客户邮件有时没有明确的问题，需要 AI 自己判断类型。

**修复**：增加邮件类型识别步骤：

```python
prompt_v4 = """
你是公司客服，需要回复客户的邮件。

【客户邮件】
[邮件内容]

【处理流程】
第一步：判断邮件类型
- 投诉（产品/服务问题）
- 咨询（问问题）
- 请求（要求某种帮助）
- 反馈（建议或表扬）

第二步：根据类型选择回复策略
- 投诉：致歉 + 调查 + 解决方案 + 补偿
- 咨询：直接回答 + 相关建议
- 请求：评估可行性 + 回复 + 如需内部审批说明
- 反馈：感谢 + 如是建议说明后续计划

【公司政策】
...

【回复要求】
...

【输出格式】
---
邮件类型：[识别出的类型]
回复策略：[选择的策略]
正文：[邮件正文]
---
"""
```

### 最终版本 Prompt

```python
final_prompt = """
<role>
你是一位专业、友好的客服代表，擅长用同理心沟通，有效解决客户问题。
</role>

<task>
根据客户邮件，生成一封专业、友好的回复邮件。
</task>

<customer_email>
[客户邮件内容]
</customer_email>

<company_policy>
【可提供的解决方案】
- 退款（购买后7天内，且未使用核心功能）
- 更换产品/服务
- 优惠券（最高100元）
- 技术支持和故障排除

【不可做出的承诺】
- 超出政策范围的补偿
- 法律责任的承认
- 未经授权的折扣或免费服务

【必须升级的情况】
- 涉及金额超过1000元
- 涉及法律问题或潜在诉讼
- 客户明确表示将向监管部门投诉
- 高价值客户（年消费 > 10000元）的投诉
</company_policy>

<response_guidelines>
1. 称呼客户（使用客户姓名或"尊敬的客户"）
2. 感谢客户反馈
3. 确认理解的问题
4. 真诚道歉（如是公司原因）
5. 解释情况（如适用）
6. 提供具体解决方案
7. 说明我们做了什么防止问题再次发生
8. 询问是否满意，提供联系方式
9. 签名：公司客服团队
</response_guidelines>

<output_format>
请按以下 XML 格式输出：

<email_type>投诉/咨询/请求/反馈</email_type>
<strategy>选择的主要回复策略</strategy>
<action_taken>我们采取或即将采取的具体行动</action_taken>
<email_body>
[完整邮件正文]
</email_body>

<escalation_needed>是/否（是否需要升级）</escalation_needed>
<escalation_reason>如果需要升级，说明原因和应该升级给谁</escalation_reason>
</output_format>

<constraints>
- 邮件正文不超过 300 字
- 使用温暖但不专业的语气
- 不要使用"非常抱歉给您带来困扰"这类套话，要具体说明我们做了什么
- 不要在邮件中提及公司的具体政策条款
- 如果不确定如何回复，请说明需要进一步核实，不要编造
</constraints>
"""
```

## 案例二：代码审查 Prompt

### 需求背景

团队需要统一 Code Review 的标准。目前的问题：
- 不同人审查标准不一
- 审查意见缺乏建设性
- 漏掉常见问题

目标：做一个 Prompt，帮助进行结构化的代码审查。

### 分析代码审查的关键维度

```
1. 功能正确性：代码逻辑是否正确处理所有情况
2. 性能：是否有明显的性能问题
3. 安全：是否有安全漏洞
4. 可维护性：代码是否清晰、可读、可扩展
5. 测试：是否有足够的测试覆盖
6. 规范：是否符合团队代码规范
```

### 设计 Prompt

```python
code_review_prompt = """
<role>
你是一位资深软件工程师，有10年开发经验，专注于代码质量和安全。
你做过Google和Amazon的技术面试官，擅长发现代码中的潜在问题。
</role>

<code_to_review>
[代码内容]
</code_to_review>

<language>Python</language>

<review_focus>
请从以下维度审查代码：

1. 功能正确性
   - 逻辑是否正确
   - 边界情况是否处理
   - 是否有潜在的bug

2. 性能
   - 时间复杂度是否最优
   - 是否有不必要的重复计算
   - IO操作是否优化

3. 安全
   - 是否有注入风险
   - 敏感数据是否暴露
   - 权限检查是否完整

4. 可读性
   - 变量/函数命名是否清晰
   - 复杂逻辑是否有注释
   - 是否有重复代码可以提取

5. 测试
   - 是否有测试覆盖
   - 测试用例是否充分
   - 边界情况是否有测试

6. 最佳实践
   - 是否符合语言/框架的最佳实践
   - 是否有更优雅的实现方式
</review_focus>

<output_format>
请按以下格式输出审查结果：

## 总体评价
[代码的整体质量评估，1-10分]

## 问题列表
| 严重程度 | 位置 | 问题描述 | 建议修复方式 |
|---------|------|---------|------------|
| 🔴严重 | [代码行] | [描述] | [建议] |
| 🟡中等 | [代码行] | [描述] | [建议] |
| 🟢建议 | [代码行] | [描述] | [建议] |

## 特别亮点
[代码中做得好的地方，值得继续保持]

## 总结
[1-3句话的总结性评价]
</output_format>

<constraints>
- 每个问题都必须具体（指出代码行号或代码片段）
- 每个问题都必须有可执行的修复建议
- 区分：必须修复的问题（严重影响功能/安全/性能）vs 建议改进的问题
- 如果代码质量很好，也要指出亮点，不只是挑刺
- 不要做语义层面的重新设计，如无必要不要建议重写
</constraints>
"""
```

### 测试与迭代

用几个真实代码片段测试：

**测试 1**：一个明显有 SQL 注入风险的代码
→ 审查应该发现安全漏洞

**测试 2**：一段高质量的算法代码
→ 审查应该给出正面评价，不应该过度挑剔

**测试 3**：一段有逻辑 bug 和代码规范问题的代码
→ 审查应该两者都指出

根据测试结果微调 Prompt：
- 增加"亮点"部分，避免只会批评
- 明确"不建议重写，除非有严重设计缺陷"
- 增加代码行号的引用要求

## 实战经验总结

### Prompt 设计的常见模式

| 场景 | 推荐结构 |
|------|---------|
| 分类+处理 | 先分类 → 再根据类型选择处理方式 |
| 分析+建议 | 提供数据 → 分析维度 → 结论 → 建议 |
| 写作+审核 | 生成 → 自查清单 → 修订 → 输出 |
| 问答+追问 | 回答 → 追问确认 → 深入 → 总结 |
| 多版本生成 | 生成3个版本 → 对比 → 选择或合并 |

### 实战中的避坑指南

1. **不要假设 AI 知道你的业务**
所有业务相关的信息（产品、用户、政策）都要明确提供。

2. **不要只测试正常情况**
边界情况和错误输入的测试集，才能暴露 Prompt 的真正问题。

3. **不要一次改太多**
每次只改一个地方，才能知道哪个改法有效。

4. **不要忽视输出格式**
结构化的输出格式，不仅是给人看的，也是帮助 AI 更准确生成的手段。

5. **不要忽略错误处理**
Prompt 里要包含"遇到以下情况怎么处理"，而不是假设 AI 能正确处理所有情况。
