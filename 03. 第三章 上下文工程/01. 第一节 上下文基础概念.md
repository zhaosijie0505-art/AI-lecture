# 第一节 上下文基础概念

## 上下文是什么

理解上下文工程，先要搞清楚"上下文"在 AI 系统中到底指什么。

很多人把上下文简单理解为"对话历史"。但实际上，AI 的上下文远比对话历史丰富——它包括你给 AI 的所有信息输入，以及 AI 之前生成的所有内容。

---

## 上下文的组成

AI 的上下文（Context）由以下几部分组成：

```
┌──────────────────────────────────────────────────────┐
│ 上下文窗口                                            │
│                                                      │
│ ┌────────────────────────────────────────────────┐  │
│ │ 1. 系统提示（System Prompt）                     │  │
│ │    - 角色设定                                   │  │
│ │    - 长期行为规则                               │  │
│ │    - 公司/产品信息                              │  │
│ └────────────────────────────────────────────────┘  │
│                                                      │
│ ┌────────────────────────────────────────────────┐  │
│ │ 2. 对话历史（Conversation History）              │  │
│ │    - 之前的用户消息                             │  │
│ │    - AI 的回复                                 │  │
│ │    - 包含截至当前的所有轮次                     │  │
│ └────────────────────────────────────────────────┘  │
│                                                      │
│ ┌────────────────────────────────────────────────┐  │
│ │ 3. 当前输入（Current Input）                   │  │
│ │    - 用户最新的一条消息                        │  │
│ │    - 可能包含附件/图片/文件                    │  │
│ └────────────────────────────────────────────────┘  │
│                                                      │
│ ┌────────────────────────────────────────────────┐  │
│ │ 4. 外部上下文（External Context）              │  │
│ │    - RAG 检索到的文档片段                      │  │
│ │    - Agent 的记忆存储                          │  │
│ │    - 工具返回的结果                            │  │
│ └────────────────────────────────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 系统提示（System Prompt）

系统提示是最先进入上下文的内容，具有"全局设定"的地位。它影响 AI 在整个对话中的行为模式。

系统提示适合放：
- 角色定义（"你是一位资深数据分析师"）
- 长期行为规则（"所有分析必须基于提供的数据"）
- 公司/产品的基础信息
- 通用知识（"我们公司成立于2018年，产品主要面向B端用户"）

系统提示的问题：
- 内容过多会被"稀释"，关键规则可能被忽略
- 内容过长会增加 Token 消耗
- 有些规则无法用自然语言准确描述

### 对话历史（Conversation History）

对话历史是"短期上下文"。它让 AI 知道你们聊了什么，从而能够：
- 理解后续问题的指代（"这个"、"它"、"上面提到的"）
- 记住之前的结论和决策
- 在之前的基础上继续深入

对话历史的挑战：
- 会话越长，历史越长，Token 消耗越大
- 需要在"记住"和"忘记"之间做取舍
- 错误的历史信息会"污染"后续回答

### 当前输入（Current Input）

当前输入是"此时此刻"用户发送的消息。它是上下文的核心——用户当前想要完成的任务、提出的问题、提供的资料都在这里。

当前输入的优化空间：
- 信息是否完整？
- 结构是否清晰？
- 是否包含了对任务必要的信息？

### 外部上下文（External Context）

外部上下文是"超越对话的上下文"。它包括：
- RAG（检索增强生成）系统从向量数据库中检索到的相关文档
- Agent 的长期记忆存储
- 工具调用返回的结果（如搜索结果、代码执行结果）

外部上下文让 AI 能够访问"对话之外"的信息，是构建真正智能应用的关键。

## Attention 机制在上下文中的作用

### 什么是 Attention

在第一节提到过，Attention 机制让 AI 理解输入文本中每个词与其他词的关系。但 Attention 的作用不止于此——它同样决定了 AI 如何利用上下文中的不同信息片段。

当上下文包含多个信息片段时，Attention 机制会在每个生成步骤中，计算应该"关注"上下文中哪些部分：

```
生成 Token 1: [系统提示] ← 关注 30%  [对话历史] ← 关注 20%  [当前输入] ← 关注 45%  [外部上下文] ← 关注 5%
生成 Token 2: [系统提示] ← 关注 25%  [对话历史] ← 关注 25%  [当前输入] ← 关注 35%  [外部上下文] ← 关注 15%
...
```

Attention 的权重是动态的——不同任务、不同生成阶段，关注的重点不同。

### 位置效应与注意力衰减

研究和对齐表明，AI 对上下文中不同位置的注意力并不均匀：

**开头和结尾的内容通常被关注更多，中间内容容易被"忽略"。**

这意味着：
- 把最重要的信息放在 Prompt 的开头或结尾
- 避免在大量中间内容中埋没关键信息
- 长上下文需要特别设计，确保重要信息不会被稀释

```
位置:   [开头]  ████████████████████████████  [中间]  ░░░░░░░░░░░░░░░░  [结尾]  ████████████████████████████
        ←──────────────── 高 ────────────────→  ←── 低 ──→  ←───────────── 高 ──────────────→
```

### Attention 对上下文工程的影响

理解 Attention 机制，给我们以下启示：

**1. 信息要精炼，不要堆砌**
上下文越长，有效信息被"摊薄"得越厉害。4000 Token 的精炼上下文，可能比 10000 Token 的冗长上下文效果更好。

**2. 重要信息要突出**
可以用格式（列表、标签）标记重要内容，让 Attention 机制更有效地识别。

**3. 结构化优于自然段落**
结构化的内容（列表、表格）比连续的自然段落更容易被 Attention 识别和提取。

**4. 相关信息要靠近**
如果两条信息相关，把它们放在一起，而非分开放在上下文的两端。

## 上下文窗口的限制

### 什么是上下文窗口

上下文窗口（Context Window）是 AI 一次能处理的最大 Token 数量。超出这个范围的文本会被截断。

主流模型的上下文窗口：

| 模型 | 上下文窗口 |
|------|-----------|
| Claude 3.5 Sonnet | 200K Tokens |
| Claude 3 Opus | 200K Tokens |
| GPT-4 Turbo | 128K Tokens |
| Gemini 1.5 Pro | 1M Tokens |

### 截断的后果

当上下文超过窗口限制时，超出的部分会被截断（Truncation）。截断的内容 AI 完全看不到。

常见的截断问题：
- 对话历史过长，新消息被截断 → AI 失去对上下文的理解
- 文档过长，结尾部分被截断 → 结论/总结部分丢失
- 检索结果过多，首尾被截断 → 关键信息丢失

### 上下文窗口的实际限制

虽然 Claude 有 200K Token 的上下文窗口，但这不意味着可以随意使用：

**1. 成本因素**
按 Token 计费，上下文越长，费用越高。长上下文可能让简单任务变得很贵。

**2. 性能因素**
虽然 Claude 长上下文表现不错，但超长上下文仍然会带来：
- 响应速度下降
- 中间部分内容注意力不足
- 生成质量不稳定

**3. 工程因素**
处理长上下文需要更复杂的系统设计（分块、检索、滑动窗口），增加了开发和维护成本。

**最优策略：上下文够用就好，不要为了"充分"而塞入过多信息。**

## 上下文的类型

### 静态上下文 vs 动态上下文

| 类型 | 说明 | 示例 |
|------|------|------|
| 静态上下文 | 在对话开始前就已确定，基本不变 | 系统提示、产品基础信息 |
| 动态上下文 | 在对话过程中动态生成和更新 | 对话历史、RAG 检索结果、工具返回 |

### 短期上下文 vs 长期上下文

| 类型 | 说明 | 生命周期 |
|------|------|----------|
| 短期上下文 | 与当前任务直接相关 | 当前对话 / 当前任务 |
| 长期上下文 | 跨对话、跨任务的信息 | 用户偏好、项目背景、知识库 |

### 内部上下文 vs 外部上下文

| 类型 | 说明 | 获取方式 |
|------|------|----------|
| 内部上下文 | 直接在 Prompt 中提供 | Prompt 组成 |
| 外部上下文 | 通过工具/系统注入 | RAG、记忆系统、API 调用 |

---

## 本节小结

理解上下文基础的三个核心要点：

**1. 上下文由四部分组成**
系统提示 + 对话历史 + 当前输入 + 外部上下文。不同部分有不同特点和约束。

**2. Attention 机制决定 AI 如何利用上下文**
AI 对不同位置的内容注意力不均——开头和结尾被关注更多，中间容易被忽略。

**3. 上下文窗口有硬限制**
超出窗口的内容会被截断，重要信息不要放在可能被截断的位置。

下一节我们将学习如何管理上下文窗口——当信息量超过窗口限制时，应该如何取舍和组织。
