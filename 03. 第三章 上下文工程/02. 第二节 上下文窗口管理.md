# 第二节 上下文窗口管理

## 当上下文超载时

上一节我们理解了上下文的基本概念。但实际应用中，经常遇到的问题是：**上下文太长，塞不进去怎么办？**

这一节我们学习上下文窗口管理的核心策略——如何在有限的空间内，让 AI 获得最需要的信息。

---

## 信息优先级评估

上下文管理的本质是**信息取舍**。面对超长的信息，我们首先要学会判断：**哪些必须保留，哪些可以舍弃？**

### 评估维度

判断一条信息是否必要，可以从三个维度评估：

**1. 相关性（Relevance）**
这条信息与当前任务有多大的直接关系？

- 直接相关：必须保留（如用户明确提到的问题、数据、约束）
- 间接相关：可以精简或压缩（如背景介绍、次要细节）
- 不相关：应该删除（如无关的附加信息）

**2. 不可替代性（Irreplaceability）**
这条信息如果丢失，AI 是否还能正确完成任务？

- 高不可替代性：必须保留（如唯一的关键数据点）
- 中等：可以压缩
- 低不可替代性：可以删除

**3. 可推断性（Inferrability）**
这条信息是否可以从其他信息中推断出来？

- 不可推断：必须保留
- 可以推断：可以删除或简化

### 评估矩阵

| 相关性 | 不可替代性 | 行动 |
|--------|-----------|------|
| 高 | 高 | **必须保留** |
| 高 | 低 | **精简保留** |
| 低 | 高 | **评估是否真正必要** |
| 低 | 低 | **删除** |

## 信息压缩技术

### 1. 摘要压缩

把长文本压缩成短摘要，只保留核心信息。

```python
# 原始信息（500字）
"2024年Q3，我们的产品在华北地区的销售额达到1200万元，
同比增长25%，环比增长8%。其中，北京市场贡献最大，
达到500万元，天津市场150万元，石家庄市场100万元...
[中间省略200字]...
华东地区销售额800万元，同比增长15%..."

# 压缩后（约50字）
"2024年Q3：华北1200万元（同比+25%），北京500万领跑；
华东800万元（同比+15%）。"
```

**什么时候用**：
- 详细信息对当前任务不重要，趋势和结论才重要
- 需要保留多个数据点，但空间有限
- 原始信息有明确的结构和关键数据

**怎么压缩**：
- 提取数字和百分比
- 保留趋势描述（增长/下降/持平）
- 删除中间细节和例外情况

### 2. 结构压缩

把自然语言描述转换为结构化格式。

```python
# 原始信息（自然段落）
"张三是一名高级软件工程师，2020年加入公司，
在产品开发部门工作。他主要负责后端开发，
精通Python和Go语言，有5年以上开发经验。"

# 压缩后（结构化）
姓名：张三 | 职位：高级软件工程师 | 入职：2020年
部门：产品开发部 | 方向：后端开发 | 技能：Python, Go | 经验：5年+
```

结构化格式的优势：
- 同样信息量，Token 更少
- AI 更容易精确定位和提取
- 便于后续处理和比较

### 3. 删除停用词

在中文中，有些词对语义贡献极低，但占用 Token：

常见可删除的词：
- 语气词：可能、大概、也许、似乎
- 模糊限定词：非常、特别、相当、极其
- 重复表达：的的确确、确确实实、实实在在

**注意**：这个技巧要慎用。过度删除会让文本不自然，影响 AI 理解。

### 4. 引用而非全文

如果上下文中已经有相关信息，可以删除冗余的重复，改为"引用"已提供的信息。

```python
# 原来
prompt = """
根据以下产品信息写文案：

产品A：
- 品牌：XX
- 价格：199元
- 功能：防水、防尘、防摔
- 适用场景：户外运动

产品B：
- 品牌：XX（同上）
- 价格：299元
- 功能：防水、防尘、防摔（功能同上）
- 适用场景：户外运动（场景同上）
"""

# 优化后
prompt = """
根据以下产品信息写文案：

产品A：品牌XX，价格199元，功能：防水/防尘/防摔，场景：户外运动

产品B：品牌XX，价格299元，功能：防水/防尘/防摔，场景：户外运动
注：品牌、功能、场景与产品A相同
"""
```

## 对话历史管理

对话历史是上下文超载的最常见原因。随着对话进行，历史越来越长，占用空间越来越大，但不修剪又会让 AI 失去上下文。

### 策略一：总结压缩

当对话历史过长时，用 AI 把它压缩成摘要，后续对话只保留摘要而非完整历史。

```python
# 对话进行到一定长度后，调用压缩 Prompt
compression_prompt = """
请将以下对话历史压缩成摘要。

【对话历史】
[完整对话内容...]

【压缩要求】
1. 保留所有关键决策和结论
2. 保留用户提出的关键需求和约束
3. 保留重要的信息点（如数字、日期、名称）
4. 删除重复的解释和确认性对话
5. 用简洁的语言，不超过原文的20%

【输出格式】
## 对话摘要
[压缩后的摘要]
"""
```

压缩后的摘要：
```
用户正在为其电商项目选型。用户倾向使用Python+FastAPI。
已讨论了方案A（自建微服务）和方案B（Serverless）。
用户最后选择了Serverless方案，但担心冷启动延迟。
需要进一步讨论：成本估算、监控方案。
```

### 策略二：选择性保留

不保留完整历史，只保留最近 N 轮对话 + 关键结论摘要。

```python
# 保留策略
keep_history = [
    {"role": "user", "content": "[最近1-2轮的用户消息]"},
    {"role": "assistant", "content": "[最近1-2轮的AI回复]"},
]

# 关键结论单独存储
key_conclusions = """
1. 用户选择了 Serverless 架构
2. 主要技术栈：Python + FastAPI + AWS Lambda
3. 项目时间线：2个月内上线MVP
4. 待解决：冷启动延迟问题
"""
```

### 策略三：定期重置

对于特别长的任务型对话，可以定期"重置"上下文，用新的对话处理新阶段。

```python
# 在新的任务阶段开始时
prompt = """
【背景回顾】
- 项目：电商后端重构
- 已完成：技术选型、架构设计
- 当前阶段：数据库详细设计

【本次任务】
基于已确认的架构设计，完成数据库表结构设计。
"""
```

## 外部上下文管理

当使用 RAG 等系统注入外部文档时，同样面临上下文管理的问题。

### 分批注入

不要一次注入所有检索结果，而是根据需要分批提供。

```python
# 策略：先问 AI 需要什么信息
prompt = """
我有一个问题需要你回答：[用户问题]

同时，我有一份相关文档可以提供给你参考。
在阅读文档之前，请先告诉我：
1. 你觉得回答这个问题需要什么信息？
2. 我应该从文档中提取哪些部分给你？

这样我可以更有针对性地提供上下文。
"""
```

### 动态上下文注入

只注入与当前生成步骤相关的内容，而非全部上下文。

```python
# Agent 处理多步任务时的动态上下文
step_1_context = "只提供与步骤1相关的文档片段"
step_2_context = "只提供与步骤2相关的文档片段"
...
```

### 上下文优先级队列

当空间有限时，按优先级选择注入哪些上下文。

```
优先级 1（必选）：用户最新消息的核心相关文档
优先级 2（推荐）：上一轮对话中 AI 明确要求的信息
优先级 3（可选）：相关的背景知识或参考文档
优先级 4（删除）：过时的、不确定的、与任务无关的信息
```

## 滑动窗口技术

滑动窗口（Sliding Window）是一种处理长文档的经典技术。

### 基本原理

```
文档：[段落1][段落2][段落3][段落4][段落5][段落6][段落7][段落8]

窗口大小：3个段落
步长：2个段落

窗口1 → [段落1][段落2][段落3]
窗口2 →           [段落3][段落4][段落5]
窗口3 →                     [段落5][段落6][段落7]
...
```

每个窗口单独处理，最后合并结果。

### 在 AI 应用中的实现

```python
def process_long_document(doc, window_size=3, step=2):
    """处理长文档，每次取窗口大小的段落"""
    paragraphs = doc.split("\n\n")
    results = []

    for i in range(0, len(paragraphs), step):
        window = paragraphs[i:i+window_size]
        if len(window) < 2:  # 不足2个段落，跳过
            continue

        context = "\n\n".join(window)

        # 让 AI 分析这个窗口
        analysis = analyze_with_ai(context)

        results.append({
            "window_id": i // step,
            "paragraphs": [f"段落{i+1}" for i in range(i, min(i+window_size, len(paragraphs)))],
            "analysis": analysis
        })

    return results

def merge_results(results):
    """合并多个窗口的分析结果"""
    merge_prompt = f"""
    以下是对同一文档不同窗口的分析结果，请合并成一份完整报告：

    {results}

    【要求】
    1. 整合所有窗口的关键发现
    2. 消除矛盾点（如有）
    3. 按逻辑顺序组织
    4. 标注每个发现来自哪个窗口
    """
    return generate_with_ai(merge_prompt)
```

### 重叠窗口避免上下文断裂

滑动窗口的一个问题是窗口之间可能断裂——段落3和段落4在同一个窗口内，段落5却在下一个窗口。

**解决方式**：使用重叠窗口，确保相邻窗口有重叠内容：

```
窗口1：[段落1][段落2][段落3]
重叠 → [段落3]
窗口2：[段落3][段落4][段落5]
重叠 → [段落5]
窗口3：[段落5][段落6][段落7]
```

重叠部分提供了上下文连续性。

## 本节小结

上下文窗口管理的核心方法：

| 技术 | 适用场景 | 效果 |
|------|---------|------|
| 信息优先级评估 | 所有场景 | 决定保留什么、删除什么 |
| 摘要压缩 | 长文本、数报表 | Token 大幅减少，保持核心信息 |
| 结构压缩 | 自然段落 | Token 减少，提升 AI 提取效率 |
| 对话历史管理 | 长对话 | 防止历史膨胀导致截断 |
| 分批注入 | RAG/外部文档 | 避免无关信息稀释 |
| 滑动窗口 | 超长文档 | 处理超出窗口的文档 |

**核心原则**：上下文够用就好——精炼的信息比冗长的信息更有效。
