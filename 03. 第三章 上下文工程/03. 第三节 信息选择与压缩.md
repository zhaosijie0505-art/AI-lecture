# 第三节 信息选择与压缩

## 上下文工程的精髓

前两节我们理解了上下文是什么，以及上下文窗口管理的宏观策略。这一节我们深入学习具体的信息选择与压缩技术——怎么判断信息是否重要，怎么把长信息变短，怎么在有限空间中塞入最多有效信息。

---

## 信息的价值评估

### 相关性判断

判断一条信息是否值得保留，首要标准是**相关性**。但"相关"是一个程度问题，不是非黑即白的。

```python
# 场景：帮用户写一封商务邮件

# 高度相关信息（保留）
"收件人是财务总监，对预算审批有最终决定权"
"预算上限是50万元"
"项目计划Q3启动"

# 中度相关信息（精简）
"收件人叫张三，在公司工作5年" → "收件人：张三（财务总监）"
"邮件要发送给李四和王五" → "抄送：张三（主送）、李四、王五"

# 不相关信息（删除）
"公司去年营收增长了10%"
"办公地点在望京SOHO"
"使用的是MacBook Pro电脑"
```

### 情境依赖的相关性

信息的相关性是**情境依赖**的。同一条信息，在不同任务中，相关性可能完全不同。

```
信息："明天的天气预报是下雨"

相关性评估：
- 任务1：给客户发邮件告知会议改期 → 高度相关（解释改期原因）
- 任务2：分析公司季度财报 → 无关
- 任务3：写一篇旅游攻略 → 可能相关（提供穿衣建议）
```

**启示**：在评估信息相关性时，必须结合具体任务和场景。

### 信息的时效性

有些信息只在特定时间窗口内有价值，过时就变成噪音。

```python
# 高时效性（立即使用，过时删除）
"今天是2024年3月15日"
"当前用户在线状态：活跃"
"本次会议的议题"

# 低时效性（长期有效）
"公司成立于2018年"
"产品的核心功能：数据分析"
"客服邮箱：support@example.com"
```

## 信息压缩技术

### 1. 关键点提取（Key Point Extraction）

从长文本中提取最重要的几个点，丢弃细节。

```python
# 原始文本（会议记录，500字）
"今天下午2点，我们开了产品评审会。
参会人员有产品经理张华、技术负责人李明、设计负责人王丽。
会议主要讨论了下季度产品路线图。
张华首先介绍了Q4的功能规划，包括A、B、C三个大功能。
李明对技术可行性提出了担忧，特别是A功能需要新的算法支持，
可能需要3个月的开发周期。王丽从用户体验角度，建议优先做B功能，
因为用户调研显示B功能需求更高。

经过讨论，我们初步决定：
1. B功能优先级最高，Q4第一个月开始开发
2. A功能作为储备项目，等技术验证后再启动
3. C功能暂缓，下次会议再讨论。

下次会议定在3月20日，主要议题是B功能的详细设计。"

# 压缩后（关键点，100字）
"产品评审会结论：
- B功能优先（用户需求高，技术可行）
- A功能储备（需3个月研发期）
- C功能暂缓
下次会议：3月20日（B功能详细设计）"
```

### 2. 结构化转换（Structural Transform）

把自然语言转换为表格、列表、键值对等结构。

```python
# 原始：自然语言产品描述
"我们公司叫做星辰科技，成立于2020年5月，
总部在北京。我们主要做企业级SaaS产品，
核心产品是一款数据分析平台。我们的客户主要是中大型企业，
覆盖金融、医疗、零售等行业。目前团队有50人，
其中技术团队30人，产品团队10人，市场团队10人。
去年的年收入是2000万元，今年目标收入是5000万元。"

# 转换后：结构化
公司：星辰科技
成立：2020年5月
总部：北京
产品：企业级数据分析SaaS平台
客户：中大型企业（金融/医疗/零售）
团队规模：50人（技术30/产品10/市场10）
收入：2023年 2000万 → 2024目标 5000万
```

### 3. 公式化表达（Formulaic Expression）

用简洁的公式替代冗长描述。

```python
# 原始
"如果用户购买数量在100件以上，并且是金牌会员，
那么可以享受8折优惠；如果用户购买数量在100件以上，
但不是金牌会员，可以享受9折优惠；如果用户购买数量在100件以下，
无论是否是金牌会员，都只能享受原价，没有折扣。"

# 公式化
折扣规则：
- 数量≥100 且 金牌会员 → 8折
- 数量≥100 且 非金牌会员 → 9折
- 数量<100 → 不打折
```

### 4. 锚点-详情分离

把信息分成"锚点"（概述）和"详情"（详细说明），只在需要时展开详情。

```python
# 不分离（全部上下文）
prompt = """
项目背景：这是一个电商平台重构项目
项目目标：提升系统稳定性和扩展性
项目周期：6个月
项目预算：500万元
项目团队：20人
技术栈：Python, Go, Kubernetes, MySQL, Redis
[中间省略大量细节...]
"""

# 锚点-详情分离
prompt = """
【项目锚点】
- 项目：电商平台重构
- 目标：稳定性 + 扩展性
- 周期：6个月 | 预算：500万 | 团队：20人
- 技术栈：Python/Go + K8s + MySQL/Redis

【详情（需要时展开）】
[详细的技术方案、团队分工、里程碑等]
"""
```

### 5. 省略已知信息

如果某些信息已经通过系统 Prompt 或之前对话建立了，不需要在每次上下文中重复。

```python
# 系统提示（一次性建立）
system = "你是公司产品顾问，公司叫星辰科技，主要产品是一款B端数据分析SaaS。"

# 对话中（不需要重复公司信息）
user = "我们的竞品最近推出了一个新功能，你怎么看？"
# AI 知道"我们"指的是星辰科技，不需要再说明
```

## 上下文注入时机

### 预注入 vs 按需注入

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| 预注入 | 在对话开始前就把所有可能需要的信息都放进上下文 | 上下文有限、任务明确 |
| 按需注入 | 等 AI 要求时再提供对应信息 | 信息量大、不确定需要哪些 |

### 按需注入的实现

```python
# AI 判断需要什么信息
prompt = """
用户问题：[用户的问题]

我已经提供了以下信息：
- [已提供的信息列表]

请告诉我：
1. 回答这个问题还需要什么信息？
2. 如果需要我提供信息，请具体说明需要什么。

如果你认为已有信息足够，请直接回答问题。
"""

# 根据 AI 的回复，决定提供什么
if ai_needs_more_info:
    info_to_provide = extract_requested_info(ai_response)
    provide_context(info_to_provide)
else:
    # AI 已经可以直接回答
    display_answer(ai_response)
```

### 渐进式上下文注入

对于复杂任务，采用渐进式——先给基础上下文，看 AI 的反应，再补充更多信息。

```python
# 第一轮：最核心的信息
prompt = """
用户需求：[用户描述的核心需求]
"""

# 第二轮：根据 AI 问题补充
if ai_asks_specific_question:
    prompt += f"""
    补充信息：[AI 问的具体问题相关的内容]
    """

# 第三轮：继续补充
if ai_asks_again:
    prompt += f"""
    更多补充：[进一步相关信息]
    """
```

**优点**：避免一次性塞入过多信息；让 AI 引导上下文补充的方向。

**缺点**：需要多轮交互，速度可能较慢。

## 信息质量的判断标准

好的上下文信息应该满足：

| 标准 | 说明 | 自检问题 |
|------|------|---------|
| **准确** | 信息是正确的，没有错误 | 这个数字/事实对吗？ |
| **完整** | 提供了任务所需的全部必要信息 | AI 能从这个信息推出结论吗？ |
| **简洁** | 没有冗余的词汇和重复 | 这句话有废话可以删吗？ |
| **结构化** | 信息组织有序，便于提取 | AI 能一眼看到关键点吗？ |
| **时效** | 信息是最新的，没有过时 | 这个信息现在是有效的吗？ |
| **相关** | 只包含与任务相关的内容 | 这条信息对回答问题有贡献吗？ |

## 压缩的边界

压缩不是越多越好。过度压缩会丢失关键信息。

### 哪些不能压缩

**1. 具体数字和日期**
"用户增长了20%" vs "用户增长了"——百分比不能省。
"3月15日" vs "最近"——具体日期不能抽象化。

**2. 专业术语和专有名词**
如果这些词是任务的必要组成部分，不能用通俗表达替代。

**3. 直接引用**
如果用户提供了具体文字，尽量保留原文而非转述——原文更准确。

**4. 边界条件和例外**
"除以下情况外：[列表]"——这个列表不能压缩成"有少数例外"。

### 压缩检查清单

压缩前问自己：
- [ ] 这个信息删除后，AI 还能正确完成任务吗？
- [ ] 这个细节对结论有影响吗？
- [ ] 用户是否可能需要这个细节？
- [ ] 压缩后的表达是否准确传递了原意？

如果任何答案是"是"，就不要压缩。

---

## 本节小结

信息选择与压缩的核心原则：

**1. 评估信息的三个维度**
相关性（与任务的关系）、不可替代性（能否从其他信息推断）、时效性（是否过期）

**2. 五种压缩技术**
关键点提取 / 结构化转换 / 公式化表达 / 锚点-详情分离 / 省略已知信息

**3. 注入时机：预注入 vs 按需注入**
按需注入更节省上下文空间，但需要多轮交互

**4. 压缩有边界**
数字、日期、专业术语、边界条件不能压缩——它们可能是关键。

下一节我们将学习分块策略——当文档太长时，怎么拆、怎么拼才能让 AI 准确理解。
