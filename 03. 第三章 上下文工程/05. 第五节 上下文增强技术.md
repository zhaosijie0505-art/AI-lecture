# 第五节 上下文增强技术

## 让 AI 访问更多信息

有时候，AI 需要的信息既不在系统 Prompt 中，也不在对话历史中，而是在"外部"——比如企业的知识库、用户的个人文档、或者实时更新的数据。

上下文增强技术解决的问题是：**如何让 AI 在需要时，动态地获取这些外部信息，并注入到上下文中。**

---

## RAG：检索增强生成

### 什么是 RAG

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将信息检索与语言生成结合的技术。它让 AI 能够从外部知识库中检索相关信息，然后基于这些信息生成回答。

```
用户问题 → 检索 → 相关文档片段 → 注入上下文 → AI 生成回答
```

### RAG 的工作流程

```
1. 索引阶段（Indexing）
   文档 → 分块 → 向量化 → 存储到向量数据库

2. 检索阶段（Retrieval）
   用户问题 → 向量化 → 在向量数据库中搜索最相关的块

3. 生成阶段（Generation）
   用户问题 + 检索到的块 → 注入上下文 → AI 生成回答
```

### RAG 的核心价值

**1. 实时性**
知识库可以随时更新，AI 回答的是最新信息，而非训练数据中的旧知识。

**2. 可控性**
你可以精确控制 AI 能看到什么信息，适合企业场景的信息安全要求。

**3. 可解释性**
AI 的回答基于哪些文档，都可以追溯。

**4. 成本效益**
相比 Fine-tuning（微调），RAG 更新知识的成本极低——只需更新知识库，无需重新训练模型。

### RAG 的局限性

**1. 检索质量依赖**
如果检索不到正确的内容，AI 就无法给出正确答案。

**2. 上下文窗口限制**
即使检索到很多相关内容，能注入上下文的仍然有限。

**3. 幻觉风险**
AI 仍然可能在检索内容的基础上"添油加醋"。

## 向量检索基础

### 什么是向量检索

向量检索（Vector Search）是用"语义相似度"而非"关键词匹配"来搜索内容的技术。

```python
# 传统搜索（关键词匹配）
query = "苹果的营养价值"
只返回包含"苹果"和"营养价值"关键词的文档

# 向量搜索（语义理解）
query = "苹果的营养价值"
返回讨论水果营养成分的文档，即使不包含"苹果"这个词
```

### Embedding 模型

把文本转换为向量（Embedding）的模型叫 Embedding 模型。

主流 Embedding 模型：

| 模型 | 向量维度 | 特点 |
|------|---------|------|
| text-embedding-3-large (OpenAI) | 3072 | 效果好，但商业使用有成本 |
| text-embedding-3-small (OpenAI) | 1536 | 性价比高 |
| BGE-large-zh (智源) | 1024 | 中文效果好，开源 |
| Voyage-3 (VoyageAI) | 1024 | 中文效果好 |

### 向量数据库

存储向量和进行相似度搜索的数据库：

| 数据库 | 特点 | 适用场景 |
|-------|------|---------|
| ChromaDB | 轻量级，易用，本地优先 | 原型、快速验证 |
| FAISS | Facebook 开源，召回性能高 | 大规模向量检索 |
| Pinecone | 云服务，托管简单 | 生产环境，快速上线 |
| Qdrant | 开源，支持混合检索 | 需要中文+混合检索 |
| Weaviate | 功能丰富，支持混合检索 | 复杂查询需求 |

## 记忆系统

### 为什么 Agent 需要记忆

Agent 区别于简单问答的关键之一，是它能够**跨对话记住用户的信息和偏好**。这种能力通过记忆系统实现。

```python
# 没有记忆的对话
用户：我的名字叫张三
AI：好的，张三你好！

用户2（开启新对话）：我叫李四
AI：你好，李四！

用户2：上次我叫啥？
AI：抱歉，我没有上次对话的记忆。
```

```python
# 有记忆的对话
用户：我的名字叫张三，在星辰科技工作
AI：好的，张三，在星辰科技工作。已记住这些信息。

用户2（开启新对话）：我叫李四
AI：你好，李四！

用户2：我叫什么名字？在哪工作？
AI：根据我记录的信息，你叫李四。但上次对话中，一位叫张三的用户提到他在星辰科技工作——那是你吗？

用户2：不是，那是另一个人
AI：好的，已更新。请问你叫什么名字在哪里工作？
```

### 记忆的类型

| 类型 | 说明 | 生命周期 | 实现方式 |
|------|------|---------|---------|
| 工作记忆 | 当前任务进行中的信息 | 当前任务 | 直接在 Prompt 中 |
| 对话记忆 | 本次对话的历史 | 当前对话 | 对话历史管理 |
| 用户画像 | 用户的长期偏好和信息 | 长期 | 持久化存储 |
| 知识记忆 | 项目、团队、业务的背景 | 长期 | 知识库 |
| 技能记忆 | Agent 学会的技能和规则 | 长期 | 技能系统 |

### 记忆的读写

**写入记忆**：
```python
def save_to_memory(fact, memory_type="user_profile"):
    """将信息写入记忆"""
    memory_store.add(
        content=fact,
        type=memory_type,
        timestamp=datetime.now(),
        importance=evaluate_importance(fact)  # 评估重要性
    )
```

**读取记忆**：
```python
def retrieve_memory(query):
    """根据查询检索相关记忆"""
    # 向量检索
    results = memory_store.search(query, top_k=5)

    # 按重要性过滤
    filtered = [r for r in results if r.importance > threshold]

    # 格式化输出
    return format_memories(filtered)
```

### 记忆的遗忘

Agent 不需要记住所有事情。记忆系统需要"遗忘"机制：

```python
def cleanup_memory():
```

## 混合检索

单一检索方式往往不够好。**混合检索**结合多种检索方法，弥补各自的不足。

### 向量检索 + 关键词检索

```python
def hybrid_search(query, top_k=5):
    """
    混合检索：向量 + 关键词
    """
    # 向量检索结果
    vector_results = vector_db.search(query, top_k=10)

    # BM25 关键词检索结果
    bm25_results = bm25_index.search(query, top_k=10)

    # RRF 融合（Reciprocal Rank Fusion）
    # 对两个结果列表的排名进行融合
    fused = rrf_fusion([
        (vector_results, 0.6),  # 向量检索权重
        (bm25_results, 0.4)     # BM25权重
    ], top_k)

    return fused
```

### 混合检索的原理

RRF（Reciprocal Rank Fusion）融合公式：

```
Score(d) = Σ 1/(k + rank_i(d))

其中：
- d 是文档
- k 是常数（通常 60）
- rank_i(d) 是文档 d 在第 i 个结果列表中的排名
```

排名越靠前的文档，融合分数越高。

### 其他混合方式

```python
# 稀疏检索（关键词）+ 密集检索（向量）
hybrid_results = sparse_dense_fusion(
    sparse_results,  # BM25、TF-IDF 等
    dense_results    # 向量检索结果
)

# 多跳检索：基于初步结果扩展查询
```

## 上下文增强的最佳实践

### 1. 检索时机

不要在用户每次提问时都检索所有知识库。根据问题类型决定是否检索：

```python
def should_retrieve(query, has_memories):
    """
    判断是否需要检索
    """
    # 明确需要检索的场景
    if contains_knowledge_request(query):  # "根据XX文档..."
        return True

    # 明确不需要检索的场景
    if is_general_knowledge(query):  # "今天天气怎么样"
        return False

    # 有记忆且相关
    if has_memories and is_relevant_to_memory(query):
        return True

    # 默认检索（保守策略）
    return True
```

### 2. 检索结果排序

检索到的结果需要排序，把最好的放在最前面：

```python
def rank_retrieved_docs(docs, query):
    """
    多维度排序
    """
    scored = []
    for doc in docs:
        score = {
            "relevance": calculate_relevance(doc, query),      # 与查询的相关性
            "completeness": evaluate_completeness(doc),        # 内容的完整度
            "freshness": calculate_freshness(doc),            # 时效性
            "quality": doc.quality_score                       # 文档本身的质量
        }
        # 加权求和
        final_score = (
            score["relevance"] * 0.4 +
            score["completeness"] * 0.2 +
            score["freshness"] * 0.2 +
            score["quality"] * 0.2
        )
        scored.append((final_score, doc))

    # 按分数排序
    scored.sort(reverse=True)
    return [doc for _, doc in scored]
```

### 3. 上下文组装

把检索结果组装成上下文时，需要处理好多个文档的关系：

```python
def assemble_context(query, retrieved_docs, max_tokens=4000):
    """
    将检索结果组装成上下文
    """
    context_parts = []
    current_tokens = 0

    for doc in retrieved_docs:
        doc_tokens = count_tokens(doc.content)

        if current_tokens + doc_tokens > max_tokens:
            # 检查剩余空间是否还能容纳摘要
            remaining = max_tokens - current_tokens
            if remaining > 200:
                context_parts.append(f"[文档摘要] {summarize(doc, remaining)}")
            break

        # 添加完整文档
        context_parts.append(f"[来源：{doc.source}] {doc.content}")
        current_tokens += doc_tokens

    # 添加引用说明
    if context_parts:
        context_parts.append(
            f"\n[以上内容来自 {len(retrieved_docs)} 个文档片段，"
            "请基于这些内容回答问题。]"
        )

    return "\n\n".join(context_parts)
```

## 本节小结

上下文增强的三大技术：

**1. RAG（检索增强生成）**
索引 → 检索 → 生成。适合让 AI 访问外部知识库。关键是检索质量——检索不到就生成不了。

**2. 记忆系统**
工作记忆 / 对话记忆 / 用户画像 / 知识记忆 / 技能记忆。Agent 需要在合适的时机读写记忆，并主动遗忘不重要的信息。

**3. 混合检索**
向量检索 + 关键词检索 + 多跳检索。单一检索方式有局限，混合检索能显著提升召回质量。
