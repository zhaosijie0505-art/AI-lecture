# 第一节 Harness 与评估框架概述

## 为什么 AI 系统评估如此困难

评估 AI 系统之所以难，是因为 AI 的"正确"不是非黑即白的。传统软件的 bug 是明确的——程序崩溃、返回错误结果、除零错误。但 AI 的问题往往是模糊的：

```
问：什么是量子计算？
答1："量子计算是一种利用量子力学原理进行信息处理的计算方式..."
答2："量子计算是一种前沿的计算技术，通过量子比特来执行运算..."
答3："量子计算是一种基于量子力学原理的革命性计算范式..."

三个答案都正确，但表述不同。哪个更好？好多少？
```

这种模糊性使得 AI 评估不能简单地用 `assert output == expected` 来解决。我们需要一套新的评估范式。

---

## AI 评估的两大类

AI 评估分为两大类：**内置评估**和**独立评估**。

### 内置评估（Intrinsic Evaluation）

直接评估模型生成的内容本身。比如：
- 这个句子语法正确吗？
- 这段文字流畅吗？
- 这个回答是否偏离主题？

内置评估的问题：**不知道什么是"正确答案"**。

### 外置评估（Extrinsic Evaluation）

评估模型在实际任务中的表现。比如：
- 回答了用户的问题吗？
- 完成了用户要求的任务吗？
- 对业务指标有正向影响吗？

外置评估的问题：**需要真实用户反馈，成本高、周期长**。

实践中，两者需要结合使用——内置评估快速筛选，外置评估最终验证。

## 四大评估维度

无论用什么框架，AI 评估通常围绕四个核心维度展开：

### 1. 能力维度（Capability）

模型"会不会"——模型是否具备完成特定任务的知识和技能。

- **知识储备**：是否知道正确答案
- **推理能力**：是否能正确推理
- **语言能力**：语法、流畅性、多语言

常见 Benchmark：
- MMLU（57个学科的知识测试）
- BIG-bench（200+任务综合测试）
- GSM8K（数学应用题）
- HumanEval（代码生成）

### 2. 对齐维度（Alignment）

模型"听不听话"——模型输出是否符合人类期望和指令要求。

- **指令遵循**：是否按要求执行
- **安全性**：是否生成有害内容
- **真实性**：是否产生幻觉
- **有帮助性**：是否真正有帮助

常见 Benchmark：
- TruthfulQA（真实性）
- RealToxicityPrompts（毒性检测）
- BBQ（偏见测试）
- AlpacaEval（指令遵循）

### 3. 实用维度（Practicality）

模型"能不能用"——在实际部署场景中的表现。

- **延迟**：响应速度是否可接受
- **成本**：推理成本是否可承受
- **稳定性**：输出是否一致
- **可扩展性**：能否处理高并发

这类指标通常不在标准 Benchmark 中，需要自己设计。

### 4. 用户体验维度（User Experience）

模型"好不好用"——用户实际使用感受。

- **有用性**：回答是否有帮助
- **清晰度**：表达是否清楚
- **适当性**：语气是否合适
- **信任度**：用户是否信任回答

这类指标通常需要人类评估（Human Evaluation）。

## 选型指南：什么场景用什么框架

| 场景 | 推荐框架 | 理由 |
|------|---------|------|
| 评估 RAG 系统整体效果 | RAGAS / Trulens | 专门为 RAG 设计，指标全面 |
| 评估通用 LLM 能力 | BIG-bench / HELM | 任务覆盖全面，学术界认可 |
| 评估对话系统 | MT-Bench / ChatArena | 支持多轮对话和人类偏好 |
| 评估中文模型 | CMMLU / C-Eval | 中文语境，学科覆盖全 |
| 检测偏见/幻觉 | Giskard / TruthfulQA | 专注安全性和真实性 |
| 企业内部应用评估 | Giskard + 自定义指标 | 灵活性高，可集成 |
| 快速原型验证 | AlpacaEval | 简单易用，30分钟跑完 |

## 评估框架的局限性

任何框架都有局限性，使用时需要心中有数：

### 1. Benchmark 饱和

当模型在某个 Benchmark 上得分接近满分时，这个 Benchmark 就不再有区分度。

```
GPT-3（2020）：MMLU ~43%
GPT-4（2023）：MMLU ~86%
GPT-4o（2024）：MMLU ~88%

86% → 88%，这2%的提升有多大意义？
```

### 2. Benchmark 过拟合

模型开发者可能会针对 Benchmark 优化，导致 Benchmark 失去公正性。

### 3. 任务覆盖不完整

再全面的 Benchmark 也不可能覆盖所有真实场景。Benchmark 得分高，不代表在实际应用中表现好。

### 4. 指标与体验脱节

自动指标（BLEU、ROUGE）和人类对"好"的判断往往不一致。

**最佳实践**：
- 用 Benchmark 作为**快速筛选**工具
- 用人类评估作为**最终验证**手段
- 结合业务指标（转化率、满意度）作为**实际效果**衡量

---

## 本节小结

AI 评估的难点在于输出的概率性和多样性，传统的确定性测试方法不适用。

AI 评估分为**能力、对齐、实用、体验**四个维度，每个维度需要不同的指标和评估方法。

主流框架有通用 LLM 评估（BIG-bench/HELM）、对话评估（MT-Bench）、RAG 评估（RAGAS/Trulens）、应用层评估（Giskard）四大类。

任何框架都有局限性——Benchmark 是筛选工具，不是最终答案。结合人类评估和业务指标才能全面判断 AI 系统的真实效果。
