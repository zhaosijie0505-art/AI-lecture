# 第五节 微调 vs RAG 决策框架

## 两条技术路线的本质区别

在构建 AI 应用时，最常见的选择是：**微调（Fine-tuning）** 还是 **RAG（检索增强生成）**？这两个技术路线解决的是不同的问题，适合不同的场景。

先理解它们的本质区别：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   微调（Fine-tuning）                                           │
│   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                     │
│   本质：改变模型本身                                            │
│                                                                 │
│   预训练模型 ──→ [喂大量特定领域数据] ──→ 微调后模型            │
│                                    ↑                            │
│                            模型的"知识"和"行为"                  │
│                            被永久改变                           │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   RAG（检索增强生成）                                           │
│   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                     │
│   本质：不改变模型，只改变输入的上下文                          │
│                                                                 │
│   模型 ──→ [检索相关文档] ──→ 注入上下文 ──→ 生成回答           │
│                           ↑                                     │
│                     外部知识库                                  │
│                     随时可更新                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

## 微调的适用场景

### 什么时候选择微调

**1. 需要改变模型的"行为模式"时**
当你想让模型按照特定的方式回答，而不是按照"通用"的方式：

```python
# 例子：医疗场景的对话风格微调
# 微调前（通用ChatGPT）：
# "肺癌的治疗方法主要包括手术、放疗、化疗..."

# 微调后（医疗专用）：
# "针对非小细胞肺癌，我院的标准化治疗路径如下：\n
# 1. I期-IIIA期：优先考虑手术切除...\n
# 2. IIIB期-IV期：以系统性治疗为主..."

# 微调让模型：
# - 严格按医疗指南格式输出
# - 始终包含"请咨询专业医生"的风险提示
# - 不会给出超出指南的建议
```

**2. 需要模型"学会"特定领域思维时**
当领域有独特的推理模式或术语体系：

```python
# 法律场景：微调让模型学会"法律思维"
# 问题："房东提前终止合同怎么办？"

# 微调前（通用回答）：
# "可以协商解决，或者根据合同条款处理..."

# 微调后（法律思维）：
# "根据《民法典》第563条，有下列情形之一的，当事人可以解除合同：
# （一）因不可抗力致使不能实现合同目的...
# 具体到您的情况，建议按以下步骤操作：\n
# 1. 审查合同中的提前终止条款...
# 2. 书面通知对方..."
```

**3. 需要特定格式或结构化输出时**
当输出必须是特定格式（JSON、表格、特定标记）：

```python
# 微调让模型稳定输出 JSON
# 问题：提取用户信息

# 微调前：输出可能是自然语言
# 微调后：100% 按 JSON 格式输出
{
    "name": "张三",
    "age": 28,
    "occupation": "工程师",
    "skills": ["Python", "Go", "K8s"]
}
```

### 微调的代价

| 维度 | 说明 |
|------|------|
| **成本** | 需要 GPU 资源；GPT-4 微调成本 $0.008/1K tokens |
| **时间** | 训练需要数小时到数天 |
| **数据量** | 通常需要 1000-10000 条高质量标注数据 |
| **可迭代性** | 模型行为一旦固定，更新需要重新训练 |
| **可解释性** | 模型学到的"知识"无法直接查看 |

## RAG 的适用场景

### 什么时候选择 RAG

**1. 需要实时更新知识时**
知识会频繁变化（每天、每周更新）：

```python
# 例子：产品知识库
# 产品功能、价格、库存每天都在变

# RAG 方案：
# 今天用户问："iPhone 15 现在多少钱？"
# 系统检索最新的价格文档，返回："iPhone 15 128GB 售价 5999 元"
# 明天价格变了，只需要更新文档，不需要重新训练

# 微调方案：
# 模型只知道训练时的价格
# 如果价格变动，需要重新收集数据 → 重新训练 → 等待上线
```

**2. 需要访问大量长尾知识时**
知识体量太大，无法全部装进模型：

```python
# 例子：企业知识库
# 100 万字的内部文档、2000 页的产品手册、数千个历史案例

# 微调的问题：
# - 100万字 >> 模型 context window
# - 长尾知识在微调中容易被稀释
# - 新增知识需要重新训练

# RAG 的优势：
# - 可以索引任意规模的文档
# - 检索只返回相关片段
# - 新增文档只需重建索引
```

**3. 需要可解释性和溯源时**
必须知道答案来自哪个文档：

```python
# 例子：合规审计
# "你们的回答需要能追溯到具体哪份政策文件"

# RAG 天然支持：
# "根据《用户隐私政策》第3.2条的规定：..."
# 回答 → 引用了哪份文档 → 审计人员可验证

# 微调无法提供这种溯源：
# 模型回答"根据政策"，但这个"政策"是模型内化的，
# 无法指向具体文件
```

**4. 需要多个独立知识域时**
不同场景需要不同的知识组合：

```python
# 例子：客服机器人
# 专业知识（产品说明）+ 通用能力（对话）+ 实时信息（库存）

# RAG 方案：
# 专业知识：走 RAG 检索
# 通用能力：模型自带
# 实时库存：走 API 查询
# → 按需组合，互不干扰

# 微调方案：
# 所有知识都要装进一个模型
# 不同知识域可能相互干扰
# 知识冲突难以处理
```

### RAG 的代价

| 维度 | 说明 |
|------|------|
| **检索质量** | 检索不到正确文档 → 生成质量差 |
| **延迟** | 检索 + 生成，比纯生成多一步 |
| **工程复杂度** | 需要维护向量数据库、检索系统 |
| **上下文长度** | 注入的文档仍然受 context window 限制 |

## 决策矩阵

### 选择微调的情况

```
推荐微调的特征：
✓ 需要模型的行为/语气/格式保持一致
✓ 领域有独特的推理模式（如法律、医疗）
✓ 需要模型"内化"某种能力（如代码生成、翻译）
✓ 知识相对稳定，不频繁更新
✓ 对延迟敏感（不想每次都检索）
✓ 有足够的训练数据和 GPU 资源

典型场景：
- 法律文书分析（专业格式 + 法律推理）
- 代码审查（特定代码规范 + 审查模式）
- 产品客服（固定品牌语气 + 标准回复格式）
- 对话助手（特定角色设定 + 对话风格）
```

### 选择 RAG 的情况

```
推荐 RAG 的特征：
✓ 知识需要频繁更新（每天/每周）
✓ 知识体量巨大，无法全部内化
✓ 需要答案可溯源到具体文档
✓ 知识来自多个异构来源
✓ 需要混合使用多种知识
✓ 预算有限，无法承担微调成本

典型场景：
- 企业知识库问答（文档频繁更新）
- 产品手册问答（大量长尾问题）
- 实时数据查询（价格/库存/新闻）
- 合规/审计场景（需要文档溯源）
```

### 两者结合（微调 + RAG）

```
最常见的最优解：两者结合

微调提供：基础能力 + 行为模式
RAG 提供：实时知识 + 文档溯源

例子：医疗 AI 助手
├── 微调：让模型学会医学术语、医疗推理模式、标准回复格式
│         "症状A + 症状B + 指标C → 建议做XX检查"
├── RAG：提供最新临床指南、药品说明、医保政策
│         实时检索"2024年新版《中国高血压防治指南》"
└── 结果：既有专业能力，又有最新知识
```

def assess_knowledge_characteristics():
    """
    评估你的知识特性

    # 问题 3：知识的准确性要求多高？

    # 问题 4：需要溯源吗？
    need_traceability = True

    return {

        "knowledge_size": knowledge_size,
        "accuracy_requirement": accuracy_requirement,
        "need_traceability": need_traceability
    }

### 第二步：成本收益分析

```python
def fine_tune_vs_rag_analysis(scenario):
    """
    微调 vs RAG 成本收益分析
    """
    analysis = {}

    # 微调成本估算
    fine_tune_cost = estimate_fine_tune_cost(
        dataset_size=scenario.get("labeled_data", 0),  # 标注数据量
        training_epochs=scenario.get("epochs", 3),
        model_size=scenario.get("model_size", "7B"),
        gpu_hours=scenario.get("gpu_available", 0)
    )
    analysis["fine_tune"] = {
        "cost": fine_tune_cost,
        "time": estimate_fine_tune_time(fine_tune_cost),
        "update_cost": fine_tune_cost,  # 每次更新都需要重新训练
    }

    # RAG 成本估算
    rag_cost = estimate_rag_cost(
        document_count=scenario.get("docs", 1000),
        query_volume=scenario.get("monthly_queries", 10000),
        embedding_model=scenario.get("embedding_model", "text-embedding-3-small")
    )
    analysis["rag"] = {
        "cost": rag_cost,
        "time": "1-2 days",
        "update_cost": 0,  # 只需更新文档
    }

    # 推荐
    if scenario.get("update_frequency") == "daily":
        analysis["recommendation"] = "RAG"
    elif scenario.get("labeled_data", 0) > 5000:
        analysis["recommendation"] = "Fine-tune + RAG"
    else:
        analysis["recommendation"] = "Start with RAG"

    return analysis
```

无论选择哪条路，都要先做小规模验证：

## 混合策略最佳实践

### 策略一：基础模型 + 领域微调 + RAG

```
┌─────────────────────────────────────────────────────────────┐
│  用户问题                                                     │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  领域微调模型                                                │
│  （学会领域术语、推理模式、输出格式）                          │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  RAG 检索（补充实时知识）                                     │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  最终回答（行为专业 + 知识最新）                              │
└─────────────────────────────────────────────────────────────┘
```

### 策略二：路由决定

```python
def route_query(query, routes):
    """
    根据问题类型选择不同处理路径
    """
    # 分析问题类型
    if is_factual_lookup(query):
        # 事实查询 → RAG
        return routes["rag"]

    elif is_procedural_task(query):
        # 流程性任务 → 微调模型（行为稳定）
        return routes["fine_tuned"]

    elif is_creative_task(query):
        # 创意任务 → 基础模型
        return routes["base"]

    else:
        # 混合
        context = retrieve_rag_context(query)
        return routes["fine_tuned_with_context"](context)
```

## 本节小结

微调和 RAG 解决不同问题：

**微调**：改变模型本身，适合需要稳定行为模式的场景。
**RAG**：改变输入上下文，适合需要实时更新知识的场景。

**决策树**：
- 知识频繁更新？→ RAG
- 领域有独特推理模式？→ 微调
- 需要答案溯源？→ RAG
- 知识体量巨大？→ RAG
- 有大量标注数据 + 稳定需求？→ 微调

**最优解通常是两者结合**：微调提供基础能力，RAG 提供实时知识。
