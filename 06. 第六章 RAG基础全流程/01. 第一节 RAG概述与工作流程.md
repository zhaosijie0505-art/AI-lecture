# 第一节 RAG 概述与工作流程

## RAG 的前世今生

RAG（Retrieval-Augmented Generation）概念由 Meta AI（Facebook）在 2020 年提出，最初用于解决语言模型的幻觉问题和知识时效性问题。

```
RAG 发展历程：
2020 - Facebook 发布 RAG 论文，提出检索增强生成架构
2021 - REALM 模型将知识检索融入预训练
2022 - ChatGPT 发布，RAG 成为私域知识问答的主流方案
2023 - RAG 与 LLM 结合成为企业 AI 落地标配
2024 - 高级 RAG（Self-RAG、Corrective-RAG）成熟
```

---

## RAG 是什么

**RAG = Retrieval（检索）+ Augmented（增强）+ Generation（生成）**

简单来说：让 LLM 在回答问题前，先去一个知识库里检索相关的资料，然后基于检索到的资料生成答案。

```
传统 LLM 流程：
用户 → LLM → 回答（依赖训练知识）

RAG 流程：
用户 → 检索知识库 → 找到相关文档 → 一起发给 LLM → 基于资料生成回答
```

---

## 为什么需要 RAG

### LLM 的三大局限

**1. 知识有截止日期**
```
GPT-4 的训练数据截止到 2023 年 12 月
→ 问"今天股市怎么样"，LLM 无法回答
→ 问"2024 年发布的新产品"，LLM 不知道
```

**2. 无法访问私有数据**
```
企业有大量内部数据：产品手册、客服记录、内部政策
→ 这些数据 LLM 没学过
→ LLM 回答不了"我们公司XX产品的保修期是多久"
```

**3. 容易产生幻觉**
```
问："你们公司的年假政策是什么？"
LLM 瞎编：年假是 15 天（实际是 10 天）
→ 危害极大，尤其是医疗、法律、金融领域
```

### RAG 如何解决这些问题

```
RAG = 实时知识库 + 检索 + 基于资料生成

实时知识库：更新文档即可，不用重新训练模型
检索：找到与问题最相关的资料
基于资料生成：答案有据可查，降低幻觉
```

## RAG 架构三大组件

```
┌─────────────────────────────────────────────────────────────┐
│                    RAG 架构                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────┐   ┌─────────────────┐   ┌─────────┐ │
│   │  文档处理       │ → │  向量数据库     │ → │   LLM   │ │
│   │  (Ingestion)    │   │  (Vector Store) │   │(Generate)│ │
│   └─────────────────┘   └─────────────────┘   └─────────┘ │
│          ↓                      ↑                         │
│   ┌──────────────┐      ┌──────────────┐                  │
│   │  文档加载    │      │   检索       │                  │
│   │  文本分块    │      │  (Retrieval) │                  │
│   │  Embedding   │      └──────────────┘                  │
│   └──────────────┘                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1. Ingestion（文档入库）

```
原始文档（PDF/Word/HTML/Markdown）
        ↓
文档解析（提取文本）
        ↓
文本分块（切成合适大小的段落）
        ↓
Embedding（将文本转为向量）
        ↓
存入向量数据库
```

### 2. Retrieval（检索）

```
用户问题
        ↓
Embedding（将问题转为向量）
        ↓
向量相似度搜索（在数据库中找最相似的文档块）
        ↓
返回 Top-K 个相关文档
```

### 3. Generation（生成）

```
用户问题 + 检索到的文档
        ↓
构建 Prompt（将文档作为上下文注入）
        ↓
发给 LLM 生成回答
        ↓
返回带引用的答案
```

## RAG 完整工作流程

### 步骤 1：文档准备与入库

```python
# RAG 入库流程示例
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma

# 1. 加载文档
loader = PyPDFLoader("document.pdf")
documents = loader.load()

# 2. 文本分块
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,    # 每块 1000 字符
    chunk_overlap=200   # 相邻块重叠 200 字符
)
chunks = splitter.split_documents(documents)

# 3. 生成 Embedding 并存入向量数据库
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)

print(f"成功入库 {len(chunks)} 个文档块")
```

### 步骤 2：用户查询与检索

```python
# RAG 查询流程示例

# 1. 用户提问
query = "这个产品的保修期是多久？"

# 2. 检索相关文档
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 3}  # 返回 top-3 最相关文档
)
relevant_docs = retriever.get_relevant_documents(query)

print(f"检索到 {len(relevant_docs)} 个相关文档：")
for i, doc in enumerate(relevant_docs, 1):
    print(f"\n文档 {i}: {doc.page_content[:200]}...")
```

### 步骤 3：基于检索结果生成

```python
# 3. 构建 Prompt 并生成回答
from langchain.chat_models import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

# 构建 RAG Prompt
template = """基于以下参考资料回答用户问题。如果资料不足，直接说你不知道。

参考资料：
{context}

用户问题：{question}

回答："""

prompt = ChatPromptTemplate.from_template(template)

# 调用 LLM 生成
llm = ChatOpenAI(model="gpt-4o")
chain = prompt | llm

response = chain.invoke({
    "context": "\n\n".join([doc.page_content for doc in relevant_docs]),
    "question": query
})

print(response.content)
```

## RAG 与微调的对比

### 何时选择 RAG

```
选择 RAG 的场景：
✓ 知识需要频繁更新（每天、每周更新）
✓ 需要答案可溯源到具体文档
✓ 知识体量很大（远超微调的数据量承受能力）
✓ 多个不同知识域需要混合使用
✓ 预算有限，无法承担微调成本

典型场景：
- 企业知识库问答
- 产品手册问答
- 实时数据查询
- 合规/审计场景
```

### 何时选择微调

```
选择微调的场景：
✓ 需要模型有特定的行为模式或语气
✓ 领域有独特的推理模式
✓ 需要模型的输出格式高度一致
✓ 知识相对稳定，不需要频繁更新
✓ 有足够的标注数据和计算资源

典型场景：
- 客服机器人的固定回复风格
- 法律文书分析（专业格式）
- 代码审查（特定代码规范）
```

### 两者结合（最优解）

```
RAG + 微调 的结合：

微调提供：
- 稳定的领域术语和表达方式
- 固定的任务执行模式
- 符合业务需求的输出格式

RAG 提供：
- 最新鲜的实时知识
- 可溯源的文档引用
- 大规模知识的快速检索

实际案例：医疗 AI 助手
- 微调：让模型学会医学术语、医疗推理模式
- RAG：提供最新临床指南、药品说明
```

## RAG 的评估指标

RAG 系统的质量直接影响最终回答的效果，主要评估维度：

| 指标 | 说明 | 目标 |
|------|------|------|
| **检索召回率** | 相关文档被检索到的比例 | > 90% |
| **检索精确率** | 检索到的文档中真正相关的比例 | > 80% |
| **上下文利用率** | LLM 实际使用了多少检索内容 | 越高越好 |
| **答案准确率** | 最终答案的正确程度 | > 85% |
| **引用准确率** | 引用的文档块是否真正支撑答案 | > 90% |

## 本节小结

RAG = 检索（Retrieval）+ 增强（Augmented）+ 生成（Generation）

RAG 解决 LLM 的三大局限：知识截止日期、私有数据无法访问、幻觉问题

RAG 三大组件：文档入库（Ingestion）、检索（Retrieval）、生成（Generation）

RAG vs 微调：RAG 适合知识频繁更新、需要溯源；微调适合行为模式固定、需要特定语气。两者结合效果最佳。
