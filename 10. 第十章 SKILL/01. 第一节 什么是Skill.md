# 第一节 什么是 Skill

## 从"每次都要教"到"教一次就会"

用 Claude 时，你有没有这种感觉：每次开始一个新任务，都要先花大段时间交代背景、规则和期望的输出格式？而且即使交代得很清楚，Claude 在不同 session 里的表现也可能不太一致？

传统 Prompt 的问题在于——**它是一次性的**。每次新对话都是零基础，Claude 需要从你的消息里重新理解你的偏好和规则。而 Skills 解决的是：把"你希望 Claude 怎么做"这件事，变成一种持久的能力。

Skill 的本质是：**一套可复用的专业知识、流程步骤和最佳实践的封装**。

当你告诉 Claude "这是我做财务分析的标准化流程"或者"这是我们公司的文档格式规范"，Claude 学会之后，在任何场景下都能自动应用这套规则——不需要重复提醒，不需要每次都写长 prompt，效果始终一致。

---

## Skill 的官方定义

Anthropic 官方对 Skills 的描述是：

> *"Skills package your expertise and procedures so Claude delivers expert-level output every time."*
>
> 把你的专业知识和流程打包，让 Claude 每次都能交付专家级的输出。

这个定义里有几个关键词：

**package（打包）** — Skills 不是一段 Prompt 文本，而是一个结构化的能力单元，包含触发条件、使用说明、流程步骤和输出规范。

**every time（每次）** — 关键在**一致性**。同一个 Skill 多次使用，输出质量和风格是稳定的。这解决的是"我用 Claude 做事时好时坏"的问题。

**expert-level（专家级）** — Skills 的目标是把组织里最懂行的人的方法论，复制给 AI，让 AI 也能输出接近专家水平的结果。

---

## Skill 与相关概念的区分

为了准确理解 Skill，我们把它和几个容易混淆的概念做对比：

### Skill vs. Tool

**Tool（工具）** 是 Claude 调用外部能力的接口，比如搜索、代码执行、文件读写。Tool 的调用是**模型决策**的产物——Agent 在推理过程中判断需要调用什么工具。

**Skill（技能）** 描述的是**"怎么做"**的专业知识，不是"能做什么"的能力扩展。

| 对比维度 | Tool | Skill |
|----------|------|-------|
| 回答的问题 | "Claude 能调用什么外部能力？" | "做这件事的标准流程是什么？" |
| 本质 | 能力接口 | 流程规范 |
| 调用方式 | 模型自主决策 | 场景自动匹配或手动指定 |
| 典型例子 | 搜索工具、计算器、代码执行器 | 财务报告模板、品牌文案规范、数据分析步骤 |

一个类比：**Tool 是让 Claude 长出手和脚，Skill 是让 Claude 长出经验和直觉。**

### Skill vs. Agent

**Agent** 是一个能自主规划、执行多步骤任务的智能体。Agent 关注的是**目标达成**，它会自己决定下一步做什么。

**Skill** 关注的是**方式方法**，它定义的是做某类事情的标准化流程，不涉及自主规划和执行控制。

| 对比维度 | Agent | Skill |
|----------|-------|-------|
| 关注点 | 做什么（目标） | 怎么做（方法） |
| 自主性 | 高（自主规划执行） | 低（被动遵循流程） |
| 典型场景 | "帮我完成这个项目" | "按这个模板生成报告" |
| 关系 | Agent 可以调用 Skill | Skill 被 Agent/用户调用 |

实际使用中，Agent 和 Skill 是协同关系：Agent 负责规划"做什么"，Skill 负责确保"怎么做"是专业的。

### Skill vs. RAG

**RAG（检索增强生成）** 是从文档库中检索相关知识，注入到上下文中帮助模型回答。

RAG 的本质是**知识查找**——在大量文档里找到相关信息。

Skill 的本质是**流程规范**——不仅包含知识，还包含做事的标准步骤和质量要求。

| 对比维度 | RAG | Skill |
|----------|-----|-------|
| 回答的问题 | "这个问题的答案在哪里？" | "做这件事的标准流程是什么？" |
| 知识形态 | 文档内容 | 流程 + 规范 + 示例 |
| 调用触发 | 文档相似度匹配 | 场景/任务类型匹配 |
| 效果 | 答案更准确 | 流程更标准 |

### Skill vs. Prompt Template

**Prompt Template（提示模板）** 是预设的 Prompt 格式，包含变量占位符，每次使用时填入不同内容。

Skill 和 Prompt Template 的区别是：Prompt Template 是**文本格式**的复用，Skill 是**能力**的复用。

| 对比维度 | Prompt Template | Skill |
|----------|-----------------|-------|
| 复用内容 | Prompt 文本格式 | 知识 + 流程 + 规范 |
| 灵活性 | 高（变量替换） | 中（遵循流程但可适应） |
| 学习能力 | 无（机械替换） | 有（上下文自适应） |
| 持久性 | 一次性使用 | 跨会话持久生效 |

一个 Prompt Template 像是填表，Skill 像是教会 Claude 一项技能。

---

## Skill 的核心特征

从上面的对比，我们可以总结出 Skill 区别于其他概念的三个核心特征：

### 1. 知识与流程的封装

Skill 不仅仅是"告诉 Claude 怎么做"，而是把**做某类事情的专业知识、标准化流程和最佳实践**打包成一个完整的单元。

举例：创建一个"财务分析报告"Skill，不是写一段 Prompt，而是包含：
- 财务数据的读取和校验标准
- 分析框架（如杜邦分析、现金流分析）
- 图表规范（颜色、标注、格式要求）
- 语言风格（专业程度、术语使用）
- 审核 checklist

这种封装保证了输出的**完整性和一致性**。

### 2. 场景自动匹配

用户不需要每次手动指定"使用这个 Skill"。系统会根据任务上下文**自动判断**应该调用哪个 Skill。

这解决了"我需要用但经常忘记用"的问题——Skills 就像专业顾问，Claude 遇到相关场景会自动匹配，不需要人类提醒。

### 3. 跨平台一致性

同一个 Skill 可以在 Claude.ai 网页版、Claude Code 编程工具和 API 中通用，不需要针对不同平台调整。

这对于企业场景特别重要——同一套专业标准，在不同使用场景下保持一致。

---

## 小结

Skill 不是 Tool，不是 Agent，也不是 RAG。它是一种新的抽象层次：**把专业知识封装成可复用能力的机制**。

它的核心价值在于解决"专业能力无法规模化复制"的问题——让组织里的最佳实践不再只存在于少数专家脑子里，而是变成所有人和 AI 都能调用的标准能力。
