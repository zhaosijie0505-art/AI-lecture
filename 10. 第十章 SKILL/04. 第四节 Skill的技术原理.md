# 第四节 Skill 的技术原理

## Skills 是怎么工作的

前面三节我们从概念、价值和案例认识了 Skills。这一节深入技术原理，解释 Skills 是怎么被定义、注册、调用和组合的。

虽然 Anthropic 没有完全开源 Skills 的底层协议，但通过官方文档、API 行为和行业分析，我们可以还原其核心技术逻辑。

---

## Skill 的结构

一个 Skill 不是一段文本，而是一个**结构化的能力单元**。从功能上，可以分解为以下几个组成部分：

### 1. 名称与描述（Name & Description）

每个 Skill 有唯一的名称和清晰的描述，用于：
- 在 Skill 市场中被发现和搜索
- 系统判断何时该调用这个 Skill（场景匹配）
- 用户理解 Skill 能做什么

描述的质量直接影响 Skill 的自动匹配效果。

### 2. 触发条件（Trigger）

触发条件定义了**什么时候该调用这个 Skill**，有两种典型方式：

**显式触发**：用户主动指定使用某个 Skill，比如在对话中说明"用财务分析 Skill 处理这个表格"。

**隐式触发**（自动匹配）：系统根据任务上下文自动判断应该调用哪个 Skill。比如用户上传了一个财务报表，系统自动匹配"财务分析 Skill"，不需要用户指定。

隐式触发依赖于 Skill 描述的质量和任务分类模型的判断。

### 3. 指导内容（Instruction）

这是 Skill 的核心，包含：

**流程步骤（Procedure）**
- 做这件事的标准流程是什么
- 先做什么、后做什么
- 决策点和分支条件

**质量标准（Quality Standards）**
- 输出的质量要求是什么
- 格式规范、语言风格、详细程度
- 需要避免的错误和问题

**示例（Examples）**
- 输入是什么、期望输出是什么
- 正例和反例的对比
- 边界情况的处理

### 4. 变量与参数（Variables & Parameters）

Skill 支持参数化，使得同一个 Skill 可以适应不同的具体场景：

```
{{company_name}}  - 公司名称
{{report_type}}   - 报告类型
{{date_range}}    - 数据时间范围
```

参数在调用时由用户指定或在上下文中自动提取。

## Skill 的注册与存储

### 注册流程

创建 Skill 的基本流程：

1. **编写 Skill 定义**：用 YAML 或 JSON 格式编写 Skill 的完整定义
2. **测试验证**：在沙箱环境中测试 Skill 的效果
3. **发布到 Skill 库**：发布到个人、组织或公共 Skill 市场
4. **版本管理**：Skill 更新需要版本控制，用户可以选择锁定版本

### 存储层级

Skills 支持多级存储：

**个人 Skill（Personal Skills）**
- 仅创建者可用
- 存储在用户个人空间
- 用于个人工作偏好和流程

**组织 Skill（Organization Skills）**
- 团队或公司内共享
- 集中管理和审核
- 保证团队标准一致性

**市场 Skill（Marketplace Skills）**
- Anthropic 官方或第三方发布
- 公开可用，经过审核
- 最佳实践和专业领域包

企业场景下，组织 Skill 是核心——公司可以建立自己的 Skill 库，把专业能力集中管理，确保全公司使用同一套标准。

## Skill 的调用机制

### 调用判断

当用户发起一个任务时，系统判断是否需要调用 Skill、以及调用哪个 Skill 的过程：

1. **任务解析**：理解用户想做什么（意图识别）
2. **Skill 匹配**：根据任务类型和上下文，匹配相关 Skill
3. **参数提取**：从对话中提取 Skill 需要参数
4. **执行 Skill**：把 Skill 的指导内容注入到模型上下文中
5. **结果整合**：Skill 输出与模型能力结合，生成最终结果

这个过程对用户是透明的——用户感知到的是"AI 自动理解了我的需求，正确地处理了"。

### 上下文注入

Skill 的内容最终以**系统 Prompt 和 Few-shot 示例**的形式注入到模型上下文：

```
[系统角色定义]
你是一个专业的财务分析师，遵循公司财务分析报告标准。

[财务分析 Skill 指导]
在做财务分析时，请按以下步骤进行：
1. 首先校验数据完整性...
2. 使用杜邦分析框架...
...

[用户输入]
请分析这份季度财报...
```

这种注入方式确保模型在生成响应时，始终遵循 Skill 定义的标准。

## Skill 的组合（Stack）

### Stack 的概念

多个 Skill 可以组合使用，形成"Skill Stack"。这解决了复杂任务需要多领域能力配合的问题。

比如一个"商业尽职调查"任务，可能涉及：
- 财务分析 Skill
- 市场研究 Skill
- 法律合规 Skill
- 文档撰写 Skill

这四个 Skill 可以组成 Stack，Claude 会根据任务进展自动判断在什么环节调用哪个 Skill。

### 组合的执行逻辑

Stack 的执行不是简单的"把所有 Skill 的内容都塞进去"，而是：

1. **分解任务**：把复杂任务分解为子任务
2. **匹配 Skill**：每个子任务匹配对应的 Skill
3. **顺序执行**：按正确顺序调用 Skill
4. **上下文传递**：前一个 Skill 的输出作为后一个的输入
5. **结果整合**：整合各 Skill 的输出形成最终结果

用户不需要知道这些细节，只需要描述目标，Claude 自动完成整个流程。

## Skill 与 MCP 的关系

MCP（Model Context Protocol，我们第九章详细讲过）是连接 AI 与外部工具和数据的协议。Skill 和 MCP 是**互补的关系**：

- **MCP** 提供"**能做什么**"的能力（访问文件、调用 API、操作软件）
- **Skill** 提供"**怎么做**"的知识（流程、规范、标准）

举例：MCP 让 Claude 能够读取 Excel 文件，Skill 让 Claude 按照公司规范分析这个 Excel 文件。两者结合，Claude 才能完整地完成一项专业工作。

这个关系解释了为什么 Skills 通常和 MCP 配合使用——MCP 是手脚，Skill 是经验和直觉。

## Skill 的生态

### Marketplace

Anthropic 建立了 Skill 市场（Marketplace），用户可以：
- 发现和安装他人创建的 Skill
- 发布自己的 Skill 供他人使用
- 评价和评论 Skill 的质量

Marketplace 让 Skills 的生态有了类似 App Store 的可能性——专业能力可以被交易和共享。

### 垂直领域 Skill 包

除了通用的个人 Skill，很多专业服务商开始提供**垂直领域的 Skill 包**：
- 法律领域：合同审查 Skill、诉讼准备 Skill
- 金融领域：投资分析 Skill、风险评估 Skill
- 医疗领域：病历整理 Skill、药物相互作用检查 Skill
- 制造领域：质量检测 Skill、工艺优化 Skill

这些 Skill 包不是通用模板，而是经过该领域专家验证的专业流程。

---

## 小结

Skills 的技术原理可以总结为：

| 组件 | 功能 | 技术形式 |
|------|------|---------|
| 名称描述 | Skill 的身份和发现 | 元数据 |
| 触发条件 | 何时调用 | 匹配规则 |
| 指导内容 | Skill 的核心能力 | 系统 Prompt + 示例 |
| 参数变量 | 适应不同场景 | 模板变量 |
| Stack | 多 Skill 协作 | 任务分解 + 顺序调用 |

Skills 的设计体现了几个工程原则：
- **声明式**：描述"做什么"和"怎么做"，而不是写死流程
- **可组合**：单一 Skill 可组合，复杂任务自动分解
- **可观测**：Skill 的行为可追踪和评估
- **可演进**：Skill 可以更新迭代，用户选择版本
