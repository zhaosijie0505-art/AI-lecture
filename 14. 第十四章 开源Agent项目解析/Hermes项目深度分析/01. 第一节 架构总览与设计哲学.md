# 第一节：架构总览与设计哲学 —— 从入口到工具的全景地图

![hermes-architecture-overview](image/hermes-architecture-overview.png)

## 1. 整体架构：三层分离

在动手看代码之前，我们先把 Hermes Agent 的整体架构搞清楚。它的代码有 14 万行左右，但核心逻辑集中在以下几个层次：

```
┌─────────────────────────────────────────────┐
│            用户接入层 (User Interface)        │
│  hermes (CLI)  │  hermes gateway (消息网关)  │
└────────────────────────┬────────────────────┘
                         │
┌────────────────────────▼────────────────────┐
│           AIAgent 核心引擎 (run_agent.py)     │
│         对话循环 + 工具调度 + 状态管理          │
└────────────────────────┬────────────────────┘
                         │
┌────────────────────────▼────────────────────┐
│              工具系统 (tools/)                │
│  registry │ terminal │ file │ web │ mcp │ delegate │
└────────────────────────┬────────────────────┘
                         │
┌────────────────────────▼────────────────────┐
│           支撑模块 (agent/ │ hermes_cli/)    │
│  prompt构建 │ 上下文压缩 │ CLI命令 │ 配置管理   │
└─────────────────────────────────────────────┘
```

**第一层**是用户跟你打交道的界面：终端里敲 `hermes` 进去是交互式 TUI；或者启动 `hermes gateway` 让它跑在服务器上，接入 Telegram、Discord、Slack 等平台跟人聊天。

**第二层**是真正的 AIAgent 引擎，在 `run_agent.py` 里。这层负责：大循环——把用户消息发给 LLM，处理 LLM 的响应（可能是文本、可能是工具调用），循环直到任务完成或者达到迭代上限；工具调度——决定哪个工具被调用、怎么并行；状态管理——上下文压缩、会话历史。

**第三层**是具体工具，`tools/` 目录下的各个模块，每个工具独立实现自己的功能。

**支撑模块**包括系统提示的构建（`agent/prompt_builder.py`）、上下文窗口不够时的压缩（`agent/context_compressor.py`）、CLI 的配置和命令系统（`hermes_cli/`）。

## 2. 入口文件：cli.py 和 run_agent.py 的分工

很多同学一上来就读 `run_agent.py`，这是对的，但容易忽略 `cli.py` 的重要性。实际上这两个文件是**两条完全不同的启动路径**：

```python
# cli.py — 交互式终端
class HermesCLI:
    def run(self):
        # 启动 TUI (prompt_toolkit)
        # 读取用户输入
        # 调用 AIAgent.chat()

# run_agent.py — 底层引擎
class AIAgent:
    def chat(self, message: str) -> str:
        # 调用 run_conversation() 核心循环
```

当你运行 `hermes` 命令时，流程是：
1. `cli.py` 的 `main()` 被调用（通过 `hermes_cli/main.py` 的 entry point）
2. `HermesCLI` 初始化 TUI，进入交互循环
3. 你的每条消息通过 `AIAgent.chat()` 处理
4. `AIAgent.chat()` 调用 `run_conversation()` 核心循环

如果你用 Telegram 发消息给 Hermes，流程变成：
1. `hermes gateway start` 启动 `gateway/run.py` 的 `GatewayRunner`
2. Telegram 的 webhook 把消息推过来
3. `GatewayRunner` 为每个用户会话创建一个 `AIAgent` 实例
4. 同样走 `run_conversation()` 核心循环

**关键理解**：`cli.py` 和 `gateway/` 负责的是\"怎么接收消息、怎么把结果还给你\"，真正的 AI 逻辑在 `run_agent.py` 里。这两条路径最终汇聚到同一套核心引擎，保证了体验的一致性。

## 3. 文件依赖链：理解加载顺序

```
tools/registry.py  （无依赖，工具系统的根基）
       ↑
tools/*.py         （每个工具在 import 时调用 registry.register()）
       ↑
model_tools.py    （导入所有工具，执行 discover_builtin_tools()）
       ↑
run_agent.py / cli.py / batch_runner.py / gateway/
```

这个依赖链非常重要。`tools/registry.py` 是整个工具系统的中心，所有其他模块都依赖它，但它自己没有任何外部依赖。工具模块（如 `terminal_tool.py`）通过自注册的方式把自己挂到 registry 上，不需要 registry 知道具体工具的实现细节。这就是**控制反转**—— registry 定义接口，工具负责注册自己。

当你调用 `python -c "from model_tools import get_tool_definitions"` 时，Python 会先 import `tools/registry.py`，然后依次 import 所有工具模块，触发所有 `registry.register()` 调用，此时 registry 里就拥有了所有工具的完整信息。

## 4. 核心设计哲学：五条原则

通过阅读 Hermes 的代码，我总结出它遵循的五条设计原则：

**原则一：让配置驱动行为，而不是代码硬编码**

几乎所有可变的行为都通过 `config.yaml` 配置：模型选择、工具开关、平台凭证、皮肤主题。你去读 `hermes_cli/config.py` 会发现光 `DEFAULT_CONFIG` 就有几百行。这是 production 软件和玩具项目的本质区别之一。

**原则二：工具是插件，核心是编排**

工具系统采用**自注册模式**（Self-Registration Pattern）。每个工具模块是独立的，不需要修改核心代码就能增减工具。新增一个工具只需要：写一个 `tools/your_tool.py`，在里面调用 `registry.register()`，然后把文件名加到 `toolsets.py` 的导入列表（实际上通过 glob 自动发现，不需要手动加）。这就是**开闭原则**——对扩展开放，对修改封闭。

**原则三：危险操作必须可审计**

所有有副作用的操作——删除文件、执行命令、发消息——都有**审批回调机制**（`tools/approval.py`）。即使 `rm -rf /` 这种命令，工具层会先问\"真的要我执行吗？\"，只有批准了才会继续。这种设计保证了 Agent 不会在你不知情的情况下搞破坏。

**原则四：状态必须持久化，但不能成为瓶颈**

会话历史用 SQLite 存储，但用 WAL 模式保证并发安全；用抖动重试（jitter retry）处理写锁竞争；用 FTS5 虚拟表提供全文搜索。这些都是工程上的具体选择，不是教科书上的\"数据库三范式\"。

**原则五：多实例隔离，状态不互相污染**

CLI 模式、Gateway 模式、子代理模式，**每个 AIAgent 实例都是独立的**。子代理有独立的迭代预算、独立的工具集、独立的上下文窗口。这听起来是常识，但实现起来需要很多细节：比如子代理的 `_last_resolved_tool_names` 是全局变量，就需要显式保存和恢复。

## 5. 版本演进留下的痕迹

Hermes 的代码里能看到清晰的演进痕迹。`run_agent.py` 里有个 `api_mode` 参数，支持三种模式：

```python
if self.api_mode == "anthropic_messages":
    from agent.anthropic_adapter import build_anthropic_client
elif self.api_mode == "bedrock_converse":
    self.client = None  # 使用 boto3 直接调用
else:
    # OpenAI SDK (chat completions 或 Codex responses)
```

这说明 Hermes 一开始只支持 OpenAI 的 chat completions 格式，后来演进到支持 Anthropic 的 messages API，再后来支持 AWS Bedrock。这段代码里的 `else` 分支就是历史包袱——它既处理 OpenAI chat completions，又处理 Codex，用同一个 client，但两个的 API 格式并不完全一样。

读源码时看到这种\"奇怪的一致性\"，不要急着说\"作者水平不行\"，很可能那是渐进式演进的痕迹。

---

> [!NOTE]
> Hermes 的架构核心就三件事：入口统一到 AIAgent 引擎、工具通过自注册模式成为插件、所有状态持久化但不影响性能。
