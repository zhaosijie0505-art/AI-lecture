# 第二节：run_agent.py 核心循环 —— Agent 是怎么做决策的

## 1. 什么是"Agent 循环"

理解 `run_agent.py` 是理解整个 Hermes 的关键。这个文件实现了一个**持久化事件循环**（Persistent Event Loop），是 AIAgent 的大脑。

先搞清楚一个核心问题：为什么叫"循环"？

因为 AI Agent 工作不是"一问一答"就结束了。LLM 生成的响应可能是：
- **纯文本**：用户想看的内容，直接返回
- **工具调用**：LLM 说\"我需要查个东西\"，然后 Agent 执行工具，把结果塞回给 LLM，LLM 再生成响应，这个过程可能重复很多次

所以 Agent 的工作模式是一个 while 循环：

```
用户消息
    ↓
┌─────────────────────────┐
│  LLM 思考，给出响应        │
└────────────┬────────────┘
             │
     ┌───────┴───────┐
     ↓               ↓
  纯文本?         有工具调用?
     ↓               ↓
   返回         执行工具
             把结果塞回消息历史
                   ↓
                 继续循环
```

这就是 Agent Loop（智能体循环）。

## 2. 迭代预算机制：Agent 不能无限思考

LLM 的上下文窗口是有限的，你不能让它无限循环下去。所以 Hermes 有一个**迭代预算**（IterationBudget）机制：

```python
class IterationBudget:
    def __init__(self, max_total: int):
        self.max_total = max_total      # 最多允许多少次迭代
        self._used = 0
        self._lock = threading.Lock()
    
    def consume(self) -> bool:
        """尝试消耗一次迭代。如果预算用完，返回 False。"""
        with self._lock:
            if self._used >= self.max_total:
                return False
            self._used += 1
            return True
    
    def refund(self) -> None:
        """退还一次迭代（用于 execute_code，它不消耗预算）"""
        with self._lock:
            if self._used > 0:
                self._used -= 1
```

父 Agent 默认 90 次迭代，子 Agent 默认 50 次。这个数字不是随便定的——90 次迭代大约能处理一个中等复杂度的多步骤任务（读文件→分析→写代码→测试→修复→再测试）。

特别注意 `refund()` 方法。`execute_code` 工具执行的是用户提供的 Python 脚本，这个脚本可能跑很久、可能有很多行代码，但它的"思考过程"在 Hermes 看来是**透明的**——Hermes 不需要自己推理，它只是帮用户跑一段代码。所以 `execute_code` 消耗的迭代是**可以退还的**，不让它占用本来就不宽裕的 90 次预算。

## 3. 持久化事件循环：解决 asyncio 的生命周期陷阱

Python 的 `asyncio.run()` 有一个臭名昭著的问题：**每次调用都会创建一个新的事件循环，运行完后关闭它**。如果你在异步上下文里创建了 `httpx.AsyncClient`，当事件循环关闭时，这个 client 会被垃圾回收，**但它的底层连接可能还在引用已经关闭的事件循环**，导致 "Event loop is closed" 错误。

Hermes 的解决方案是**维护一个持久化的事件循环**：

```python
_tool_loop = None          # 主线程的持久化循环
_tool_loop_lock = threading.Lock()
_worker_thread_local = threading.local()  # 每个工作线程有自己的循环
```

```python
def _get_tool_loop():
    """获取或创建当前线程的持久化事件循环"""
    if is_main_thread():
        with _tool_loop_lock:
            global _tool_loop
            if _tool_loop is None or _tool_loop.is_closed():
                _tool_loop = asyncio.new_event_loop()
                asyncio.set_event_loop(_tool_loop)
            return _tool_loop
    else:
        # 工作线程用 thread-local 存储各自的循环
        if not hasattr(_worker_thread_local, 'loop'):
            _worker_thread_local.loop = asyncio.new_event_loop()
        return _worker_thread_local.loop
```

这样就保证了一个线程始终用同一个事件循环，避免了"创建-关闭"的生命周期陷阱。

## 4. 工具并行化：不是所有工具都能并行跑

Agent 循环里有一个重要的优化：**工具并行化**。当 LLM 一次要求调用多个工具时，Hermes 会判断这些工具能否并行执行：

```python
_NEVER_PARALLEL_TOOLS = frozenset({"clarify"})
_PARALLEL_SAFE_TOOLS = frozenset({"ha_get_state", "read_file", "search_files", ...})
_PATH_SCOPED_TOOLS = frozenset({"read_file", "write_file", "patch"})
```

不能并行的工具只有 `clarify`（因为它是用户交互工具，必须等待用户回复才能继续）。

对于涉及文件路径的工具，Hermes 还会检测路径冲突：

```python
def _should_parallelize_tool_batch(tool_calls) -> bool:
    # 1. 检查是否有永远不能并行的工具
    # 2. 检查是否有路径重叠的文件操作
    # 如果有路径重叠（比如两个 read_file 读同一个文件），串行执行
```

这个设计非常务实——并行能快，但并发读写同一文件会出问题，那就老老实实串行。

## 5. 中断机制：用户想停就能停

你按 Ctrl+C，Agent 应该停下来。这个看似简单的功能实现起来相当复杂：

```python
self._interrupt_requested = False
self._interrupt_message = None
self._execution_thread_id: int | None = None

def _set_interrupt(self, message: str = None):
    """请求正在运行的 Agent 放弃当前这一轮"""
    self._interrupt_requested = True
    self._interrupt_message = message
    self._execution_thread_id = threading.current_thread().ident
```

工具执行时会定期检查这个标志：

```python
def terminal_tool(command: str, ...):
    proc = subprocess.Popen(command, shell=True)
    while proc.poll() is None:
        if is_interrupted():   # ← 定期检查中断标志
            proc.kill()
            return {"error": "Interrupted"}
        time.sleep(0.1)       # 每 0.1 秒检查一次
    return {"exit_code": proc.returncode, ...}
```

但中断有个细节问题：**主线程的循环在工作线程执行工具时无法直接干预**。所以 Hermes 用了一个技巧：设置一个 `threading.Event`，工具在循环里检查这个 Event，如果被设置就立即退出。

## 6. 主循环流程图

```
用户消息进入
       ↓
┌─────────────────────────────┐
│ 构建消息列表 (prompt_builder)│
└────────────┬────────────────┘
             ↓
┌─────────────────────────────┐
│ 检查迭代预算 (consume)       │
│ 超限? → 返回错误，结束        │
└────────────┬────────────────┘
             ↓
┌─────────────────────────────┐
│ 调用 LLM API                │
│ (自动适配 OpenAI/Anthropic) │
└────────────┬────────────────┘
             ↓
     ┌───────┴───────┐
     ↓               ↓
  纯文本响应      有 tool_calls
     ↓               ↓
   返回用户      并行/串行执行工具
                       ↓
              ┌────────┴────────┐
              ↓                 ↓
         工具执行失败        工具执行成功
              ↓                 ↓
         记录错误到消息      工具结果加入消息历史
              ↓                 ↓
         继续循环          继续循环 (在预算内)
```

## 7. API 模式适配：为什么需要三种客户端

Hermes 支持三种 API 调用方式，写在了同一个文件里：

```python
if self.api_mode == "anthropic_messages":
    from agent.anthropic_adapter import build_anthropic_client
elif self.api_mode == "bedrock_converse":
    self.client = None
else:
    # OpenAI SDK，兼容 chat completions 和 Codex responses
```

`anthropic_messages` 模式用 Anthropic 的官方 SDK；`bedrock_converse` 用 boto3；默认的 `openai` 模式用的是 `openai` SDK 的 `chat.completions` 或 `responses` API。

为什么要分这么细？因为：
- OpenAI 的模型（GPT-4o、GPT-4o-mini）在 `chat.completions` API 上
- Anthropic 的模型（Claude 3.5、Claude 4）在 Anthropic 的 SDK/messages API 上
- AWS Bedrock 上的模型（Claude on Bedrock、Llama on Bedrock）需要用 boto3

这三种 API 的认证方式、请求格式、响应格式都不同，所以需要三套不同的客户端封装。
