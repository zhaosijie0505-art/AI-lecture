# 第三节：model_tools.py 与工具注册中心 —— 工具是怎么被发现的

## 1. 工具系统的三层架构

Hermes 的工具系统分为三层，理解这个架构能让你在debug工具问题时快速定位问题：

```
┌──────────────────────────────────────────┐
│           tools/registry.py               │  ← 工具的"数据库"
│  所有工具的注册表：schema、handler、检查函数  │
└────────────────────┬─────────────────────┘
                     │ 被 model_tools.py 查询
┌────────────────────▼─────────────────────┐
│           model_tools.py                 │  ← 工具的"编排层"
│  工具发现、过滤、调用分发、参数类型转换       │
└────────────────────┬─────────────────────┘
                     │ 被 run_agent.py 调用
┌────────────────────▼─────────────────────┐
│           tools/*.py                      │  ← 具体的工具实现
│  terminal_tool, file_tools, web_tools...  │
└──────────────────────────────────────────┘
```

---

## 2. 自注册模式：工具不需要核心代码知道它

`tools/registry.py` 实现了一个**自注册模式**（Self-Registration Pattern）。这是什么意思？

传统做法：
```python
# 核心代码需要知道所有工具
from tools.terminal_tool import terminal_tool
from tools.file_tools import read_file, write_file

# 核心代码显式管理工具列表
TOOLS = [terminal_tool, read_file, write_file, ...]
```

Hermes 的做法：
```python
# tools/terminal_tool.py 单独存在，里面这样写：
def terminal_tool(command: str, task_id: str = None) -> str:
    ...

registry.register(
    name="terminal",
    toolset="terminal",
    schema={...},  # OpenAI tool calling 格式的 schema
    handler=terminal_tool,
    check_fn=lambda: os.getenv("TERMINAL_ENV") is not None,
    requires_env=["TERMINAL_ENV"],
    is_async=False,
)
```

工具模块在 import 时自己调用 `registry.register()`，把自己挂到全局注册表上。核心代码（`model_tools.py`）只需要调用 `registry.get_definitions()` 就能拿到所有工具，不需要知道具体有哪些工具文件。

**好处**：新增一个工具不需要修改任何核心代码，只要写一个新的 `tools/your_tool.py` 文件，import 时自动被注册。

## 3. AST 静态分析：发现工具但不执行它

`tools/registry.py` 用了一种很聪明的工具发现方式——**AST 静态分析**：

```python
def _module_registers_tools(module_path: Path) -> bool:
    """通过 AST 分析模块是否在顶层调用 registry.register()"""
    source = module_path.read_text(encoding="utf-8")
    tree = ast.parse(source, filename=str(module_path))
    
    # 检查顶层语句是否有 registry.register() 调用
    return any(_is_registry_register_call(stmt) for stmt in tree.body)

def _is_registry_register_call(stmt) -> bool:
    """判断一个 AST 语句是否是 registry.register() 调用"""
    if not isinstance(stmt, ast.Expr):
        return False
    if not isinstance(stmt.value, ast.Call):
        return False
    call = stmt.value
    # 检查是否是 registry.register 或 register 调用
    func_name = _get_func_name(call.func)
    return func_name in ("register", "registry.register")
```

关键在于：它用 `ast.parse()` **只解析源码，不执行**。这避免了加载工具模块的副作用——比如某个工具文件可能依赖某个环境变量，import 时会检查环境变量，如果没设置就报错。但如果用 AST 分析，这个检查根本不会触发，因为代码根本没执行。

## 4. ToolEntry：工具的元数据结构

每个注册的工具被封装成一个 `ToolEntry`，用 `__slots__` 优化内存：

```python
class ToolEntry:
    __slots__ = (
        "name",          # 工具名称：terminal, read_file, ...
        "toolset",       # 工具集：terminal, file, web, ...
        "schema",        # OpenAI 格式的工具 schema
        "handler",       # 处理函数
        "check_fn",      # 可用性检查函数
        "requires_env",  # 需要的环变量
        "is_async",      # 是否异步
        "description",   # 描述
        "emoji",         # CLI 显示用的 emoji
        "max_result_size_chars",  # 最大结果大小
    )
```

`__slots__` 是 Python 的内存优化技巧——正常 Python 对象用 `__dict__` 存储实例属性，浪费内存；用 `__slots__` 固定属性名，内存占用能减少 30-50%。

## 5. 参数类型强制转换：解决 LLM 返回字符串的问题

LLM 返回的工具参数经常是字符串，即使 schema 里定义的是整数或布尔值：

```python
# LLM 可能返回：{"file_path": "/tmp/test.txt", "line_number": "42"}
# 实际应该：{"file_path": "/tmp/test.txt", "line_number": 42}
```

`model_tools.py` 里的 `coerce_tool_args()` 解决这个问题：

```python
def coerce_tool_args(tool_name: str, args: Dict[str, Any]) -> Dict[str, Any]:
    """按 schema 强制转换参数类型"""
    schema = registry.get_schema(tool_name)
    properties = schema.get("parameters", {}).get("properties", {})
    
    for key, value in args.items():
        if isinstance(value, str):
            expected = properties[key].get("type")
            if expected == "integer":
                args[key] = int(value)
            elif expected == "number":
                args[key] = float(value)
            elif expected == "boolean":
                args[key] = value.lower() in ("true", "1", "yes")
    return args
```

这是一个务实的设计——LLM 的 tool calling 输出格式在不同模型之间有差异，强制类型转换让上游（LLM 调用）更宽松，下游（工具执行）更健壮。

## 6. handle_function_call：工具调用的分发

当你调用 `terminal_tool` 时，流程是这样的：

```python
def handle_function_call(
    function_name: str,
    function_args: Dict[str, Any],
    task_id: Optional[str] = None,
    tool_call_id: Optional[str] = None,
    session_id: Optional[str] = None,
) -> str:
    """分发工具调用，返回 JSON 字符串"""
    
    # 1. 从注册表获取 handler
    entry = registry.get(function_name)
    if not entry:
        return json.dumps({"error": f"Unknown tool: {function_name}"})
    
    # 2. 检查工具是否可用（check_fn）
    if entry.check_fn and not entry.check_fn():
        return json.dumps({"error": f"Tool {function_name} not available"})
    
    # 3. 强制类型转换
    args = coerce_tool_args(function_name, function_args)
    
    # 4. 调用 handler（可能是同步或异步）
    if entry.is_async:
        result = _run_async(entry.handler(args, task_id=task_id, ...))
    else:
        result = entry.handler(args, task_id=task_id, ...)
    
    # 5. 截断过长的结果
    if len(result) > entry.max_result_size_chars:
        result = result[:entry.max_result_size_chars] + f"\n... [truncated {len(result) - entry.max_result_size_chars} chars]"
    
    return result
```

注意返回的是 **JSON 字符串**，不是 Python 对象。这很重要——`run_agent.py` 把工具结果添加到消息历史时，消息必须是可序列化的格式（字符串）。

## 7. 工具集（Toolsets）：按功能分组

工具不是一个大杂烩，而是按功能分成多个工具集：

```python
# toolsets.py
_HERMES_CORE_TOOLS = [
    "terminal",
    "file",
    "web",
    "delegate_task",
    "execute_code",
    "mcp",
    "session_search",
    ...
]

_TOOLSET_DEFINITIONS = {
    "terminal": {...},   # 终端工具集
    "file": {...},       # 文件工具集
    "web": {...},        # Web 工具集
    ...
}
```

用户可以通过 `hermes tools` 命令选择开启或关闭某些工具集。比如你在安全要求高的环境里，可以关闭 `terminal` 工具集，防止 Agent 执行危险命令。

---

> [!NOTE]
> 工具系统的核心是自注册模式 + AST 发现 + registry 分发三层。理解这个架构后，你就能在任何工具出问题 时快速定位：工具是否注册成功、schema 对不对、handler 是否被调用。
