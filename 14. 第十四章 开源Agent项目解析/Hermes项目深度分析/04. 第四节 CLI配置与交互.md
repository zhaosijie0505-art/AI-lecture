# 第四节：hermes_cli —— 一个专业 CLI 工具是怎么做配置管理的

## 1. CLI 的入口：main.py

`hermes_cli/main.py` 是整个 CLI 的入口点，通过 setuptools 的 entry point 机制注册为 `hermes` 命令：

```toml
# pyproject.toml
[project.scripts]
hermes = "hermes_cli.main:main"
```

当你敲 `hermes` 时，Python 找到这个 entry point，执行 `hermes_cli.main:main()`。这个函数解析命令行参数，分发到对应的子命令处理器：

```python
def main():
    parser = argparse.ArgumentParser(prog="hermes")
    subparsers = parser.add_subparsers(dest="command", required=True)
    
    # 注册子命令
    add_chat_command(subparsers)
    add_gateway_command(subparsers)
    add_setup_command(subparsers)
    add_config_command(subparsers)
    add_model_command(subparsers)
    ...
    
    args = parser.parse_args()
    args.handler(args)  # 调用对应的处理函数
```

这种用 argparse 的 subcommand 模式是标准做法，关键是**子命令的注册和分发是自动化的**——新增一个子命令只需要写一个 `add_xxx_command()` 函数注册到 subparsers，不需要修改 main() 的逻辑。

## 2. Profile 多租户：让你的配置互相隔离

Hermes 支持多 profile——每个 profile 有自己独立的配置、记忆、技能、cron 任务。这对于同时用 Hermes 做多个项目的人很有用。

```python
def _apply_profile_override() -> None:
    """在模块导入前解析 --profile/-p，动态切换 HERMES_HOME"""
    for i, arg in enumerate(argv):
        if arg in ("--profile", "-p") and i + 1 < len(argv):
            profile_name = argv[i + 1]
            os.environ["HERMES_HOME"] = f"~/.hermes/profiles/{profile_name}"
            break
```

这段代码在 **import 其他模块之前** 就执行了。为什么这么早？因为后续所有模块（`hermes_state.py`、`tools/registry.py` 等）都依赖 `HERMES_HOME` 环境变量来确定配置目录。如果你在 import 之后才设置，路径就不对了。

这个设计解决了一个实际问题：开发者想同时维护多个独立的 Hermes 配置——比如一个是个人助手，一个是工作项目助手——但部署在同一台机器上。Profile 机制让它们完全隔离。

## 3. 配置系统：YAML + 环境变量的双轨制

`hermes_cli/config.py` 定义了完整的配置系统：

```python
DEFAULT_CONFIG = {
    "model": {
        "provider": "openai",
        "name": "gpt-4o",
        "api_base": None,
    },
    "display": {
        "skin": "default",
        "tool_progress_command": True,
    },
    "tools": {
        "enabled": "all",
        "disabled": [],
    },
    ...
}
```

用户配置文件在 `~/.hermes/config.yaml`，覆盖 DEFAULT_CONFIG 中的值。

同时，`hermes_cli/config.py` 还定义了**可选环境变量**（`OPTIONAL_ENV_VARS`），用于敏感信息（如 API key）：

```python
OPTIONAL_ENV_VARS = {
    "OPENAI_API_KEY": {
        "description": "Your OpenAI API key",
        "prompt": "OpenAI API Key",
        "password": True,
        "category": "provider",
    },
    "ANTHROPIC_API_KEY": {...},
    "TELEGRAM_BOT_TOKEN": {...},
    ...
}
```

这是教科书级别的配置分层：**硬编码默认值 < YAML 用户配置 < 环境变量**。环境变量优先级最高，因为它最安全——你不会把 API key 写进配置文件然后 git push 上去。

## 4. TTY 检测保护：防止 curses 100% CPU

如果你在 CI 环境（非交互式环境）里运行 `hermes`（没有 TTY），TUI 库的初始化会进入一个不断重试的循环，CPU 占用 100%。Hermes 用 TTY 检测来防止这种情况：

```python
def _require_tty(command_name: str) -> None:
    """stdin 不是终端时退出，防止 curses 100% CPU 循环"""
    if not sys.stdin.isatty():
        print(f"Error: 'hermes {command_name}' requires an interactive terminal.")
        sys.exit(1)
```

这个保护在 `hermes chat`、`hermes tools`、`hermes config` 等需要交互界面的命令入口处都会调用。它不是写在 main() 里，而是按需分散在各个命令的处理函数里——因为有些子命令（如 `hermes gateway start`）是可以在无 TTY 环境下运行的。

## 5. 交互式 Setup 向导

`hermes setup` 是一个完整的交互式设置向导，引导用户完成：
1. 选择 LLM provider（OpenAI / Anthropic / OpenRouter / ...）
2. 输入 API key
3. 配置第一个消息平台（Telegram / Discord / ...）
4. 下载默认 Skills

```python
def run_setup_interactive():
    """引导用户一步步完成设置"""
    clear_screen()
    banner()
    
    provider = prompt_choice("Select your LLM provider:", PROVIDERS)
    api_key = prompt_password(f"Enter your {provider} API key:")
    
    store_config("model.provider", provider)
    store_config("model.api_key", api_key, secret=True)
    
    # 平台配置...
```

这里的实现有一个细节：**密码/密钥输入用 `prompt_password()` 而不是 `prompt()`**。前者不会在终端回显密码，后者会。这看起来是常识，但很多开发者做交互式 CLI 时会忘记这一点。

## 6. 皮肤引擎：数据驱动的 UI 定制

Hermes 的 CLI 界面支持换肤（skins），这是一个**纯数据驱动**的设计——不需要改代码，写一个 YAML 配置文件就能改变整个 CLI 的视觉风格。

```yaml
# ~/.hermes/skins/mytheme.yaml
name: mytheme
description: My custom theme

colors:
  banner_border: "#FF00FF"
  banner_title: "#00FFFF"
  response_border: "#FFD700"

spinner:
  thinking_faces: ["🤔", "💭", "🔍"]
  thinking_verbs: ["thinking", "analyzing", "processing"]
```

皮肤系统定义在 `hermes_cli/skin_engine.py` 里，读取用户皮肤配置后覆盖默认颜色和 emoji。内置的皮肤有 `default`（金色 kawaii 风格）、`ares`（红色战争主题）、`mono`（灰度）、`slate`（蓝色开发者主题）。

---

## 7. 皮肤引擎的数据流

```
~/.hermes/config.yaml
  display.skin: "mytheme"
          ↓
hermes_cli/skin_engine.py
  load_skin("mytheme")
          ↓
SkinConfig 数据类
  (带默认值，缺失项继承 default)
          ↓
cli.py / display.py
  用 SkinConfig 里的颜色/emoji 渲染界面
```

数据驱动的最大好处是：**不需要重启 Hermes 就能换肤**——只要 `set_active_skin("mytheme")` 就行了。

---

> [!NOTE]
> hermes_cli 的设计精髓是：命令行入口用 argparse subcommand 模式按需分发；配置用 DEFAULT_CONFIG + YAML + 环境变量三层覆盖；Profile 机制让多租户隔离；TTY 检测防止非交互环境崩溃。
