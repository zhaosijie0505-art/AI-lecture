# 第六节：hermes_state.py —— 会话是怎么被永久记住的

## 1. 为什么需要持久化存储

AIAgent 在内存里处理对话，但每次重启都会丢失之前的上下文。Hermes 用 SQLite 把会话持久化到磁盘：

```python
class SessionDB:
    """SQLite 持久化会话存储，支持 FTS5 全文搜索"""
    def __init__(self, db_path: str = None):
        self._conn = sqlite3.connect(db_path or "hermes_state.db")
        self._conn.execute("PRAGMA journal_mode=WAL")
        self._conn.execute("PRAGMA foreign_keys=ON")
```

---

## 2. WAL 模式：并发读写不打架

SQLite 默认用 rollback journal 模式——写操作会锁住整个数据库，读操作也必须等待。WAL（Write-Ahead Logging）模式解决了这个问题：

```python
self._conn.execute("PRAGMA journal_mode=WAL")
```

WAL 模式下：
- **读写可以并发**：读操作不会被写操作阻塞
- **写入使用 `BEGIN IMMEDIATE`** 获取锁：这时才真正锁定
- **checkpoint**：WAL 文件大了会合并回主数据库

这对于 Hermes 的场景非常重要——Gateway 可能同时处理多个用户的会话查询，这些查询都是读操作；在后台写入会话历史时，不应该阻塞前台的读请求。

---

## 3. 抖动重试（Jitter Retry）：解决写锁竞争

多进程同时写 SQLite 会遇到 "database is locked" 错误。Hermes 的解法是**抖动重试**：

```python
def _execute_write(self, fn: Callable[[sqlite3.Connection], T]) -> T:
    for attempt in range(self._WRITE_MAX_RETRIES):
        try:
            with self._lock:
                self._conn.execute("BEGIN IMMEDIATE")  # 获取写锁
                result = fn(self._conn)
                self._conn.commit()
                return result
        except sqlite3.OperationalError as exc:
            if "locked" in str(exc).lower():
                jitter = random.uniform(0.020, 0.150)  # 20-150ms 随机等待
                time.sleep(jitter)
                continue
    raise MaxRetriesExceeded(...)
```

抖动（jitter）的关键在于**随机化等待时间**。如果是固定等待（比如都等 100ms），多个进程会在同一时刻重试，形成" convoy effect"——大家排着队一起等，反而更堵。随机化让重试时间分散开来，大多数情况下第一个拿锁的成功，其他进程还在睡觉。

## 4. FTS5 全文搜索：让会话可以被"回忆"

用户可能问"上次我们讨论的是什么来着"，Hermes 需要在历史消息里搜索。SQLite 的 FTS5 扩展提供了高效全文搜索：

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts USING fts5(
    content,
    content=messages,      -- 关联到 messages 表
    content_rowid=id       -- 用 messages.id 作为行 ID
);

-- 触发器：messages 表新增记录时，自动更新 FTS 索引
CREATE TRIGGER IF NOT EXISTS messages_fts_insert 
AFTER INSERT ON messages BEGIN
    INSERT INTO messages_fts(rowid, content) VALUES (new.id, new.content);
END;
```

`content=messages` 的意思是 FTS 表不存储内容副本，而是**引用** messages 表的 content 列。这既提供了搜索能力，又避免了数据冗余。

## 5. 数据库 Schema：会话和消息的存储结构

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    source TEXT NOT NULL,           -- 'cli', 'telegram', 'discord'
    model TEXT,
    parent_session_id TEXT,          -- 用于子代理委托链
    started_at REAL,
    ended_at REAL,
    input_tokens INTEGER,
    output_tokens INTEGER,
    estimated_cost_usd REAL,
    summary TEXT,                   -- LLM 总结的会话摘要
    source_platform TEXT,
    source_chat_id TEXT,
    tags TEXT                        -- JSON 数组，如 ["project-a", "urgent"]
);

CREATE TABLE messages (
    id INTEGER PRIMARY KEY,
    session_id TEXT REFERENCES sessions(id),
    role TEXT NOT NULL,             -- 'user', 'assistant', 'tool'
    content TEXT,
    tool_call_id TEXT,
    tool_calls TEXT,                 -- JSON 数组
    timestamp REAL,
    reasoning TEXT,                 -- 推理过程（有些模型支持）
    sequence_num INTEGER             -- 在会话中的顺序
);
```

注意 `tool_calls` 存的是 JSON 字符串，而不是拆成单独的列——因为不同工具调用的结构差异很大，JSON 是最灵活的选择。`summary` 字段是会话压缩后留下的 LLM 总结，用于快速了解会话主题。

## 6. 用量追踪：每次 API 调用都被记录

```python
def update_session_tokens(
    self,
    session_id: str,
    input_tokens: int,
    output_tokens: int,
    model: str,
):
    """更新会话的 token 计数和估算费用"""
    cost_per_1k = {
        "gpt-4o": (2.5, 10.0),      # ($2.5/1M in, $10/1M out)
        "claude-3-5-sonnet": (3.0, 15.0),
        ...
    }
    rate = cost_per_1k.get(model, (0, 0))
    cost = (input_tokens * rate[0] + output_tokens * rate[1]) / 1_000_000
    
    self._execute_write(lambda conn: conn.execute(
        """UPDATE sessions 
           SET input_tokens = input_tokens + ?,
               output_tokens = output_tokens + ?,
               estimated_cost_usd = estimated_cost_usd + ?
           WHERE id = ?""",
        (input_tokens, output_tokens, cost, session_id)
    ))
```

这个追踪有什么用？你可以通过 `/usage` 命令查看自己在某个时间段花了多少钱、用了多少 token。这是生产环境的基础能力——没有用量追踪，月底出账单时就是一笔糊涂账。

## 7. 会话搜索：跨会话的上下文复用

```python
def search_sessions(self, query: str, limit: int = 5) -> List[dict]:
    """全文搜索历史会话"""
    results = self._conn.execute("""
        SELECT s.*, snippet(messages_fts, 0, '【', '】', '...', 32) as snippet
        FROM messages_fts
        JOIN sessions s ON s.id = messages_fts.session_id
        WHERE messages_fts MATCH ?
        ORDER BY rank
        LIMIT ?
    """, (query, limit)).fetchall()
    return [dict(r) for r in results]
```

`snippet()` 函数返回匹配上下文周围的文字片段，并用【】标记匹配的关键词。这让你在搜索时就能看到匹配的上下文，不用点进去逐条翻。
