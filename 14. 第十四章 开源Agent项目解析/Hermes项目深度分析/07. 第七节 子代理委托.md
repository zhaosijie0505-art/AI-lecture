# 第六节：delegate_tool —— 一个 Agent 是怎么把任务托付给另一个 Agent 的

## 1. 什么是子代理委托

当一个任务太复杂时（比如\"帮我分析这个代码库，写测试用例，然后部署\"），单个 Agent 循环处理所有步骤会超出迭代预算、上下文窗口也不够用。Hermes 的解法是**子代理委托**（Delegate）：父 Agent spawn 多个子 Agent，每个子 Agent 处理一个子任务。

```
用户："帮我分析这个项目，然后写测试，最后部署"
              ↓
         父 Agent
    (分解任务，分配子代理)
    /         |         \
子代理A      子代理B      子代理C
(分析)      (测试)      (部署)
   |           |           |
 结果A       结果B       结果C
    \         |         /
     父 Agent（汇总结果）
              ↓
         返回给用户
```

这就是**MapReduce 模式**在 AI Agent 里的应用。

## 2. delegate_tool 的入口

```python
def delegate_task(
    goal: str = None,
    context: str = None,
    tasks: List[dict] = None,  # 批量模式: [{"goal": "...", "context": "..."}, ...]
    toolsets: List[str] = None,
    max_iterations: int = 50,
    max_concurrent: int = 3,
    task_id: str = None,
) -> str:
```

两种调用模式：
- **单任务模式**：`goal` + `context`，spawn 一个子 Agent 处理
- **批量模式**：`tasks` 列表，spawn 最多 `max_concurrent` 个并行的子 Agent

批量模式是真正的并行执行：

```python
with ThreadPoolExecutor(max_workers=max_concurrent) as executor:
    futures = {
        executor.submit(_run_single_child, i, goal, context, ...)
        for i, goal in enumerate(batch_goals)
    }
    for future in as_completed(futures):
        results.append(future.result())
```

`as_completed()` 保证了：哪个子 Agent 先完成就先处理哪个结果，不按顺序等待。

## 3. 子代理的系统提示：专注、隔离、边界清晰

子 Agent 的系统提示跟父 Agent 不同——它需要**极度专注**，不能被父 Agent 的上下文干扰：

```python
def _build_child_system_prompt(
    goal: str,
    context: Optional[str] = None,
    workspace_path: Optional[str] = None,
) -> str:
    """为子代理构建专注的系统提示"""
    parts = [
        "You are a focused subagent working on a specific delegated task.",
        f"YOUR TASK:\n{goal}",
        "IMPORTANT CONSTRAINTS:",
        "- Work autonomously without asking the user for clarification",
        "- Report progress if the task takes more than 3 steps",
        "- Return a structured result with: what you did, what you found, any blockers",
    ]
    if context:
        parts.append(f"\nCONTEXT:\n{context}")
    return "\n".join(parts)
```

子 Agent 被明确告知：
- **干什么**：具体的 goal
- **怎么干**：自主行动，不要问用户
- **返回什么**：结构化的结果（做了什么、发现了什么、卡在哪里）

## 4. 工具限制：子代理不能做什么

子代理有独立的工具集，但不是父工具集的全部。有些工具被**禁止**：

```python
DELEGATE_BLOCKED_TOOLS = frozenset([
    "delegate_task",   # 防止递归委托——子代理再 spawn 子代理
    "clarify",         # 禁止用户交互——子代理应该自己决定
    "memory",          # 禁止写共享内存——子代理的记忆不应该污染全局记忆
    "send_message",    # 禁止跨平台副作用——子代理不能代替父 Agent 发消息
    "execute_code",    # 子代理应该逐步推理——不是一次性跑大量代码
])
```

这个列表体现了清晰的设计思路：**子代理是被严格限制的执行者，不是另一个完整的 Agent**。它只能做被分配的子任务，不能做出超出范围的决定。

## 5. 工具集继承：父代理给子代理开什么权限

子代理的工具集是父代理工具集的子集：

```python
def _build_child_agent(..., toolsets: Optional[List[str]]):
    parent_toolsets = current_agent.get_enabled_toolsets()
    
    # 去掉被禁止的，加上父代理有的
    child_toolsets = _strip_blocked_tools([
        t for t in toolsets if t in parent_toolsets
    ])
```

如果父代理没有开启 `terminal` 工具集，子代理也不能用它。这个设计保证了**权限最小化原则**——子代理只有父代理授予的权限。

## 6. 隔离的全局状态

Python 的全局变量在多线程环境里会互相干扰。`model_tools.py` 里有一个全局变量：

```python
_last_resolved_tool_names: Set[str] = set()  # 进程级全局
```

子代理执行时，这个变量会被子代理的值覆盖。Hermes 在 `delegate_tool.py` 里显式保存和恢复它：

```python
saved_tool_names = model_tools._last_resolved_tool_names
try:
    result = _run_single_child(...)
finally:
    model_tools._last_resolved_tool_names = saved_tool_names
```

这是一个典型的**保护共享状态**的模式——在线程切换时保存/恢复全局变量，防止互相污染。

## 7. 心跳保活：防止网关超时杀死子代理

子代理处理长任务时，Gateway 可能会认为 Agent \"挂了\"（因为长时间没响应）。子代理定期向父代理报告\"我还活着\"：

```python
def _heartbeat_loop():
    """定期向父代理报告活动，防止网关超时"""
    while not _heartbeat_stop.wait(_HEARTBEAT_INTERVAL):  # 每 30 秒
        if parent_agent:
            touch = getattr(parent_agent, '_touch_activity', None)
            if touch:
                desc = f"delegate_task: subagent {task_index} still working"
                touch(desc)
```

`_touch_activity()` 更新父 Agent 的最后活跃时间，阻止 Gateway 的超时机制触发。

---

> [!NOTE]
> delegate_tool 的核心是：子代理拥有隔离的上下文和受限的工具集；批量模式并行执行多个子任务；通过心跳保活防止超时；通过全局状态保存/恢复防止线程间污染。
