## 2. 设计模式： Hermes 用到的，以及为什么用

**自注册模式（Self-Registration Pattern）**
```
工具模块（import 时） → registry.register() → 全局注册表
核心代码 → registry.get_definitions() → 所有工具
```
好处：新增工具不需要改核心代码。代价：工具发现依赖约定（文件名、调用规范），调试时需要知道去哪里找。

**适配器模式（Adapter Pattern）**
```
gateway/                     → BasePlatformAdapter
         ↓
telegram.py / discord.py / slack.py（各自实现）
```
好处：一个 Agent 引擎服务所有消息平台。代价：适配器代码和平台 API 强耦合，每次平台 API 改版都要更新。

**策略模式（Strategy Pattern）**
```
get_active_env() → LocalEnvironment / DockerEnvironment / ...
```
好处：运行时根据配置切换执行策略，不需要 if-else 硬编码。代价：新增环境需要新增类。

**模板方法模式（Template Method）**
```
BasePlatformAdapter.send/receive/connect  ← 基类定义骨架
子类实现具体平台 API                    ← 具体平台差异化
```
好处的代价与适配器模式类似。

## 3. 工程细节：这些"小聪明"值得学

**抖动重试（Jitter Retry）**
```python
jitter = random.uniform(0.020, 0.150)
time.sleep(jitter)
```
解决多进程写锁竞争。固定等待时间会形成 convoy effect，随机化让重试分散开。

**`__slots__` 优化内存**
```python
class ToolEntry:
    __slots__ = ("name", "toolset", "schema", ...)
```
每个工具条目省 30-50% 内存。Agent 有几十个工具，每个工具处理成千上万次调用，累积下来可观。

**WAL 模式 SQLite**
```python
PRAGMA journal_mode=WAL
PRAGMA foreign_keys=ON
```
读不阻塞写，写用 `BEGIN IMMEDIATE` 获取锁。这是 SQLite 在多线程场景下的标准最佳实践。

**Profile 隔离在 import 前**
```python
os.environ["HERMES_HOME"] = f"~/.hermes/profiles/{profile_name}"
# 然后才 import 其他模块
```
后续所有模块都从 HERMES_HOME 读配置。时机错了配置就不生效。

**凭证脱敏**
```python
_CREDENTIAL_PATTERN.sub("[REDACTED]", error_text)
```
错误日志、用户反馈中的敏感信息自动替换。安全不是靠"不犯错"，是靠"即使错了也不泄露"。

## 4. 架构决策：这些选择不是显然的

**为什么用 SQLite 而不是 PostgreSQL？**

PostgreSQL 功能更强，但需要额外进程、需要配置连接。SQLite 是一个文件，打开就能用，部署极简。对于 Hermes 的使用场景（单用户本地数据库，多实例通过 gateway 分开），SQLite 完全够用。**够用就行，不过度设计。**

**为什么 MCP 不内置到工具系统，而是单独做成一个工具？**

因为 MCP 是**协议层**，不是功能层。MCP 服务器本身是另一个进程，Hermes 通过协议跟它通信。如果把 MCP 工具内置到每个工具里，就要处理协议解析、连接管理、类型转换等大量细节。单独抽象成 `mcp_tool.py`，保持了工具系统的简洁——你添加 MCP 工具不需要了解协议细节。

**为什么不把所有工具都并行执行？**

并行执行会冲突的是**有状态操作**，特别是文件读写。两个 `read_file` 读同一个文件，输出可能不一致；两个 `write_file` 写同一个文件，数据会损坏。Hermes 通过路径重叠检测来决定是否并行——这是务实的设计，不是理论上最优但是实现起来复杂度爆炸的方案。

## 5. 演进历史的教训

从 Hermes 的代码里你能看到演进历史：

```python
# 三种 API 模式混在一个 if-elif-else 里
if self.api_mode == "anthropic_messages":
    ...
elif self.api_mode == "bedrock_converse":
    ...
else:  # ← 包含 OpenAI chat completions 和 Codex
    ...
```

这是典型的**渐进式重构**的痕迹：
1. 一开始只支持 OpenAI chat completions
2. 后来需要支持 Anthropic，新增 `anthropic_messages` 模式
3. 再后来需要支持 Bedrock，新增 `bedrock_converse` 模式
4. 还没来得及重构老代码，就加了新功能

**不要因此嘲笑代码\"不规范\"**。这是真实项目的正常演进路径。规范的重构需要时间，但在业务压力下，边加功能边忍受技术债是常见的选择。关键是：知道哪里是技术债，未来有机会就重构。

## 6. 你在自己的项目里应该怎么做

**工具系统**：用自注册模式。不要在核心代码里写 `if tool == "terminal": do_terminal()`，用 `registry.register("terminal", handler)` 的方式。

**状态管理**：优先 SQLite，够用。 WAL 模式记得开，外键约束记得开，抖动重试记得加。

**消息平台**：适配器模式。一个 `BasePlatformAdapter`，每个平台一个子类。不要在一个文件里写 `if platform == "telegram": ... elif platform == "discord": ...`。

**危险操作**：永远加审批回调。不要假设 Agent 永远是对的，用户永远要知情。

**Profile 隔离**：在程序启动最早阶段处理路径配置，后续所有代码从同一个环境变量读配置。

---

> [!NOTE]
> Hermes 最重要的工程智慧不是某个巧妙的算法，而是**务实的取舍**：够用就行，不过度设计；渐进演进，不追求一步到位；安全冗余，不假设自己永远正确。
